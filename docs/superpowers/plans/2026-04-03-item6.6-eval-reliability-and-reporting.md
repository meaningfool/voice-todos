# Item 6.6: Eval Reliability And Reporting Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** `docs/superpowers/specs/2026-04-03-item6.6-eval-reliability-and-reporting-design.md`

**Goal:** Harden both extraction eval tracks with explicit transient-failure retries, failure-aware local artifacts, and a post-run Logfire summary download that makes speed and failure analysis available locally without repeated manual queries.

**Architecture:** Keep the existing per-experiment artifact files compact and append-only, but extend them with explicit run accounting and failure classification. Add shared eval utilities under `backend/evals/common/` for failure categorization, retry config construction, and Logfire summary generation. Both transcript-only and incremental replay runners should opt into the same retry and enrichment flow, while Logfire enrichment remains best-effort and writes one run-level `summary.json` next to the per-experiment artifacts.

**Tech Stack:** Pydantic Evals, PydanticAI retry utilities, Logfire query client, existing eval runners and artifact writer, JSON result artifacts, pytest

**References:** `backend/evals/extraction_quality/run.py`, `backend/evals/incremental_extraction_quality/run.py`, `backend/evals/extraction_quality/result_artifacts.py`, `backend/app/logfire_setup.py`, `backend/.env.example`, `backend/tests/test_extraction_runner.py`, `backend/tests/test_incremental_extraction_runner.py`, `docs/superpowers/specs/2026-04-03-item6.6-eval-reliability-and-reporting-design.md`

---

## Scope

This plan covers exactly four deliverables:

1. Shared failure classification and explicit local run accounting for both eval tracks.
2. An eval-runner retry policy for transient provider and transport failures that is separate from `--repeat`.
3. A best-effort Logfire enrichment pass that writes one run-level `summary.json` with experiment and per-case timing or failure details.
4. Runner, docs, and env-file updates so the new reporting path is understandable and testable.

Out of scope for this plan:

- changing the transcript-only or replay dataset contracts
- adding richer semantic evaluators beyond the existing count-based metrics
- changing production extraction behavior
- building a UI for these summaries
- generating `summary.md` in the first pass
- making eval success depend on Logfire availability
- per-experiment custom retry policies in the first pass

## Design Gates

### Design Gate 1 - Summary file shape

Decision:

- keep the existing per-experiment JSON artifacts as the durable base artifacts
- add one run-level sidecar file at `backend/evals/extraction_quality/results/<timestamp>/summary.json`
- make that file aggregate all experiment artifacts written in the same run directory
- include enough experiment-level and per-case data in the summary so common comparisons do not require fresh Logfire queries
- do not add a Markdown summary in the first implementation pass

### Design Gate 2 - Logfire access path

Decision:

- use the installed Logfire Python query client, not Codex MCP
- read the query token from `LOGFIRE_READ_TOKEN`
- keep `LOGFIRE_TOKEN` as the write token for shipping traces
- query Logfire by exact experiment `trace_id` from the local artifact rather than broad fuzzy filters
- if `LOGFIRE_READ_TOKEN` is missing or the query fails, write a skipped or partial summary instead of failing the eval run

### Design Gate 3 - Retry boundary

Decision:

- keep `--repeat` unchanged as repeat sampling only
- add a separate runner flag for transient task retries
- build retries with `pydantic_ai.retries.RetryConfig` passed to `Dataset.evaluate(..., retry_task=...)`
- retry only transient task failures such as provider transport, connection, or timeout failures
- do not retry evaluator failures, schema validation failures, or semantic score mismatches

```python
from collections import Counter


def classify_failure(*, error_message: str | None) -> str:
    normalized = (error_message or "").lower()
    if "modelhttperror" in normalized or "status_code: 5" in normalized:
        return "provider_transport_failure"
    if "output validation" in normalized or "unexpectedmodelbehavior" in normalized:
        return "output_validation_failure"
    if "evaluatorfailure" in normalized:
        return "evaluator_failure"
    return "unexpected_task_failure"
```

- [ ] **Step 3: Implement the shared retry helper**

Create `backend/evals/common/retry_policy.py` around `pydantic_ai.retries.RetryConfig`:

```python
from pydantic_ai.retries import RetryConfig
from tenacity import retry_if_exception, stop_after_attempt, wait_exponential


def build_retry_task_config(task_retries: int) -> RetryConfig | None:
    if task_retries <= 0:
        return None
    return RetryConfig(
        retry=retry_if_exception(is_transient_task_failure),
        stop=stop_after_attempt(task_retries + 1),
        wait=wait_exponential(multiplier=0.5, max=4),
        reraise=True,
    )
```

Implementation guidance:

- define `task_retries` as “extra attempts after the first try”
- keep the first pass global across both eval tracks
- prefer a small bounded exponential wait instead of making backoff configurable in the first pass
