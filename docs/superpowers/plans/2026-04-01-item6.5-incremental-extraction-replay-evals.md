> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** `docs/superpowers/specs/2026-04-01-item6.5-incremental-extraction-replay-evals-design.md`

**Goal:** Add a repeatable incremental replay eval harness so we can compare multiple LLM models on the same ordered extraction snapshots while each model carries forward its own previously extracted todos and is scored on the final session todo count.

**Architecture:** `main` already contains the item 6 extraction-config refactor and transcript-only eval scaffolding, including `ExtractionConfig`, config-keyed agent reuse, multi-provider model construction, experiment metadata, and JSON result artifacts. Item 6.5 should reuse those shared foundations and add only the replay-specific layers: provider-specific trace adapters that compile raw STT traces into canonical extraction checkpoints, a stable replay dataset asset, and a replay runner that threads model-produced `previous_todos` through each checkpoint sequence. We start with a Soniox trace adapter because those are the only raw traces the repo currently records, but the canonical dataset contract and replay runner should stay provider-agnostic.

**Tech Stack:** FastAPI backend code, PydanticAI, Pydantic Evals, existing `extraction_quality` eval scaffolding, Logfire, pytest, provider-specific trace fixtures (starting with Soniox), JSON dataset assets

**References:** `backend/app/extract.py`, `backend/app/model_providers.py`, `backend/app/ws.py`, `backend/app/transcript_accumulator.py`, `backend/evals/extraction_quality/experiment_configs.py`, `backend/evals/extraction_quality/result_artifacts.py`, `backend/evals/extraction_quality/run.py`, `backend/tests/test_extract.py`, `backend/tests/test_extraction_runner.py`, `backend/tests/test_replay.py`, `backend/tests/test_extraction_loop.py`, `backend/tests/fixtures/`, `docs/references/eval-models-and-stacks.md`

## Scope

This plan covers exactly four deliverables:

1. Align project metadata with the merged item 6 multi-provider eval/runtime foundations.
2. Create and maintain a dedicated replay dataset asset by compiling provider-specific traces into canonical extraction checkpoints.
3. Add a Pydantic Evals replay runner that reuses the merged item 6 experiment registry and reporting conventions while carrying forward each model's own prior todos and scoring only the final todo count.
4. Document how to run and inspect incremental replay experiments locally.

Out of scope for this plan:

- changing the production websocket extraction schedule
- storing STT trigger reasons in the replay dataset contract
- scoring intermediate steps against labeled ground truth
- final todo text reconciliation or LLM-judge scoring
- synthetic bad-prior-todo stress tests
- automatic rollout decisions

## Eval Strategy

We will treat this as an incremental extraction replay eval, not a finalized-transcript eval and not an STT eval.

- Raw seed input: provider-specific trace fixtures plus final `result.json` outputs
- Canonical eval input: deterministic `reference_dt` plus ordered extraction snapshots for a recorded session
- Replay state: `previous_todos` starts as `None` and is replaced at each step with that same model's output
- Output: structured `Todo` list at each replay step plus the final replay output
- Ground truth: one stable replay dataset asset in the repo
- Initial seed material: `backend/tests/fixtures/*/soniox.jsonl` plus `result.json`
- Initial scoring metric: `final_todo_count_match` only
- Shared experiment registry: reuse the item 6 `EXPERIMENTS` matrix so transcript-only and replayed results stay directly comparable
- Shared reporting conventions: reuse item 6 prompt metadata and JSON artifact structure where practical, extending only where replay-specific per-step outputs require it
- First experiment matrix:
  - `gemini-3-flash-preview` with provider defaults
  - `gemini-3-flash-preview` with minimal thinking
  - `gemini-3.1-flash-lite-preview` with provider defaults
  - `gemini-3.1-flash-lite-preview` with minimal thinking
  - `mistral-small-2603` with provider defaults

The initial goal is not to build a rich trajectory judge. It is to create a stable harness that exposes model-specific carry-forward behavior and answers one simple question: did the model end the replay with the right number of todos?

Important boundary:

- provider-specific logic belongs only in the trace-compilation layer
- the replay dataset contract must not contain provider-specific trigger labels
- the replay runner must consume canonical checkpoints only and must not parse raw STT traces at runtime

Experiment availability notes:

- the four Gemini experiments should work with the existing Google routing and `GEMINI_API_KEY`
- the Mistral experiment should keep the existing item 6 environment-loading pattern for `MISTRAL_API_KEY` unless a broader config cleanup is approved separately
- if the Mistral provider dependency or key is missing, the runner should skip that experiment clearly instead of failing the whole run

## Design Gates

These two design gates are now resolved for the first implementation pass:

### Design Gate 1 - Dedicated replay dataset asset

Decision:

- create one dedicated replay dataset file: `backend/evals/incremental_extraction_quality/todo_extraction_replay_v1.json`
- make that file the canonical source for replay eval runs
- treat `backend/tests/fixtures/*/soniox.jsonl` and `result.json` as seed material, not the long-term source of truth
- keep the dataset format as JSON so it stays easy to inspect and stable in git
- compile provider-specific traces into provider-neutral extraction checkpoints before writing the dataset
- store only extraction-eligible transcript snapshots in the dataset, not raw provider trace mechanics
- include `reference_dt` in every case so date-sensitive extraction stays deterministic
- keep the full expected final todo objects in the dataset even though the v1 evaluator only checks the final count

### Design Gate 2 - Evaluator strategy

Decision:

- use a single deterministic evaluator: `final_todo_count_match`
- define success as predicted final todo count exactly matching expected final todo count
- do not score intermediate steps at all in v1
- do not score final todo text, `due_date`, `assign_to`, `category`, `priority`, or `notification` in v1
- preserve per-step outputs in run artifacts or Logfire for manual inspection, but do not treat them as ground truth labels

This keeps the first replay harness intentionally narrow: it answers "did the model end this session replay with the right number of todos?" while still exposing the intermediate path for human analysis.
