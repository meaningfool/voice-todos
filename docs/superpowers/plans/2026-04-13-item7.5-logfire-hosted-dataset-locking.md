# Item 7.5 Logfire Hosted Dataset Locking Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make Logfire hosted datasets the canonical curation surface while ensuring each benchmark runs against one automatically created local lock artifact, detects dataset drift as `stale`, and requires an explicit operator choice between `--allow-stale` and `--rebase`.

**Architecture:** Keep benchmark definitions in repo YAML, but replace repo-local dataset paths with a hosted dataset reference. Introduce one active lock artifact per benchmark under `evals/locks/<benchmark_id>.json`, populated automatically on first run and rewritten on `--rebase`. Use the Logfire hosted dataset API to fetch and hash the canonical dataset, but continue using local evaluator code and the existing extraction/replay runners. Add live validation scripts that print `PASS`, `FAIL`, or `WARN` instead of silently skipping when Logfire or provider prerequisites are missing.

**Tech Stack:** Python 3.11+, Pydantic, Pydantic Evals, Logfire experimental hosted dataset API client, pytest, httpx, existing benchmark CLI and extraction/replay runners

**Spec:** `docs/superpowers/specs/2026-04-13-item7.5-logfire-hosted-dataset-locking-design.md`

**Credential reference:** `docs/references/2026-04-13-credential-storage-and-logfire-access.md`

---

## Prerequisites

- Read the full Item 7.5 spec before editing. This plan resolves the remaining implementation choices; do not improvise alternate contracts during implementation.
- Keep evaluator logic local. Hosted datasets hold case data only.
- Do not build a fake hosted dataset service just to avoid live validation. Use the real Logfire hosted dataset API for validation gates when that is what proves the behavior.
- Keep the benchmark mental model simple: `stale` means dataset hash drift only.
- Do not add runtime lock history management in this item. One benchmark has one active lock artifact.
- Follow the shared credential contract in
  `docs/references/2026-04-13-credential-storage-and-logfire-access.md`.

## Resolved Implementation Choices

- **Benchmark YAML field:** replace `dataset: <repo path>` with `hosted_dataset: <dataset_id>`.
- **Hosted identity lookup:** pass `hosted_dataset` directly to `LogfireAPIClient.get_dataset(...)` and `export_dataset(...)`. The benchmark contract uses Logfire dataset ID only.
- **Active lock path:** `evals/locks/<benchmark_id>.json`.
- **Lock file shape:** one JSON file that remains a valid dataset payload and stores lock metadata in a top-level `_benchmark_lock` field.
- **Rebase behavior:** `--rebase` rewrites the active lock file in place and reruns the full benchmark.
- **Stale override behavior:** `--allow-stale` keeps using the current active lock file even though the hosted dataset has changed.
- **Validation gate format:** every live gate script prints one of `PASS`, `FAIL`, or `WARN`. `WARN` is only for “validation could not be completed”, and must include diagnostics.
- **Credential storage:** persistent secrets live in the shared `backend/.env`;
  `.env.dev` stays non-secret and toggle-only; `backend/.logfire` remains the
  CLI-managed Logfire metadata fallback.
- **Hosted dataset credential:** use `LOGFIRE_DATASETS_TOKEN` as the persistent
  dataset-scoped API key; do not rely on the `.logfire` token having dataset
  scopes.
