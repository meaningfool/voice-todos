# Item 6.9 Benchmark Abstraction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a benchmark workflow above the existing eval experiments so benchmark authors can define benchmark manifests, attach experiments, launch experiments from a benchmark, inspect coverage, and generate stable reports from attached experiments.

**Architecture:** Add a small benchmarking subsystem under `backend/evals/benchmarking/` that owns benchmark manifests, membership updates, coverage classification, Logfire-backed reporting, and suite adapters for launching experiments through the existing eval runners. Keep experiments benchmark-agnostic by storing benchmark membership only in git-tracked benchmark manifests, and drive the implementation with automated tests derived from the operator-visible behaviors in the 6.9 spec.

**Tech Stack:** Python 3.13, argparse, pydantic-evals, Logfire, httpx, pytest, existing eval runners

**Spec:** `docs/superpowers/specs/2026-04-08-item6.9-benchmark-abstraction-design.md`

---

## Prerequisites

- Read the 6.9 spec before changing code. The operator-visible behavior changes and acceptance criteria are the required test surface for this item.
- Keep the Item 6.8 lifecycle intact. Do not reintroduce local result artifacts as a canonical store.
- Use JSON for git-tracked benchmark manifests. The repo already uses JSON for eval assets, and JSON avoids adding a new manifest dependency just for this item.
- Preserve the 6.8 meaning of `experiment_id` as the stable identity for one evaluated configuration.
- Introduce `experiment_run_id` as the unique identity for one concrete experiment execution.
- Store benchmark membership by `experiment_run_id`, not by `experiment_id` plus `batch_id`.
- Keep the benchmarking core generic across suites, but scope the initial launch integration to the existing `extraction_quality` and `incremental_extraction_quality` runners only.
- Treat manual Logfire-backed smoke validation as supplementary. The plan is not complete unless the behavior-driven automated tests exist and pass.

## Design Gate 1: Experiment run identity and benchmark attachment shape

Resolve these implementation details before coding:

- `experiment_id` stays the stable config identity
- `experiment_run_id` is added as a unique per-run identity
- benchmark manifests live under `backend/evals/benchmarks/`
- manifest format is JSON
- attached benchmark members are stored as objects, not raw strings
- minimum attached member shape:

```json
{
  "experiment_run_id": "2026-04-09T12-00-00Z-deadbeef--gemini3_flash_default",
  "note": "optional human comment"
}
```

This keeps `experiment_id` stable for configuration-level comparison while giving the benchmark layer one unambiguous reference for one concrete attached run.

The plan below assumes this decision is accepted.

```python
class AxisDefinition(BaseModel):

    values: list[str]

class AttachedExperimentRef(BaseModel):

class BenchmarkManifest(BaseModel):
    benchmark_id: str
    title: str
    description: str | None = None
    suite: str
    dataset_name: str
    dataset_sha: str
```

Create `backend/evals/benchmarking/storage.py` with:

```python
from __future__ import annotations

import json
from pathlib import Path

from evals.benchmarking.models import BenchmarkManifest, AttachedExperimentRef


def load_benchmark_manifest(path: Path) -> BenchmarkManifest:
    return BenchmarkManifest.model_validate_json(path.read_text())


def save_benchmark_manifest(path: Path, manifest: BenchmarkManifest) -> None:
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(manifest.model_dump_json(indent=2) + "\n")


def attach_experiment(
    manifest: BenchmarkManifest,
    *,
    experiment_run_id: str,
    note: str | None = None,
) -> BenchmarkManifest:
    ref = AttachedExperimentRef(
        experiment_run_id=experiment_run_id,
        note=note,
    )
    if ref not in manifest.attached_experiment_runs:
        manifest.attached_experiment_runs.append(ref)
    return manifest
```

```python
from __future__ import annotations

from itertools import product


def expected_coordinates(manifest: BenchmarkManifest) -> list[dict[str, str]]:
    fields = [axis.field for axis in manifest.axes]
    values = [axis.values for axis in manifest.axes]
    return [dict(zip(fields, combo, strict=True)) for combo in product(*values)]


def is_compatible(manifest: BenchmarkManifest, record: dict[str, str]) -> bool:
    return (
        record["suite"] == manifest.suite
        and record["dataset_sha"] == manifest.dataset_sha
        and record["evaluator_contract_sha"] == manifest.evaluator_contract_sha
        and all(record.get(key) == value for key, value in manifest.fixed_config.items())
    )
```

- maps compatible records to benchmark coordinates

## Notes for the implementing engineer

- Do not hide behavior complexity in one giant benchmark module. Keep manifest storage, coverage classification, suite launch integration, and reporting separated so the test slices stay clear.
- Prefer benchmark membership updates that are deterministic and idempotent. Re-attaching the same `experiment_run_id` should not duplicate entries.
- Keep the CLI output operator-readable. Coverage and reporting commands should make missing coordinates and incompatible attachments obvious without requiring the operator to inspect raw JSON unless they want to.
- When you need to choose between automatic discovery and stable benchmark boundaries, choose stability. This plan implements the explicit-membership design from the spec.
