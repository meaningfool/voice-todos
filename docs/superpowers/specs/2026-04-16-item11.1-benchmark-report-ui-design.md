# Item 11.1 Design: Static Benchmark Report UI

Scope: add a small benchmark-owned HTML report UI under `evals` so persisted
benchmark reports are readable in the browser without depending on the main app
frontend.

## Why this exists

The benchmark JSON reports are now durable and benchmark-owned, but they are
still raw machine-readable artifacts. They are useful for tooling and scripting,
but not pleasant to inspect by hand.

This item adds a tiny static browser UI on top of the persisted benchmark
report. It stays benchmark-generic, entry-centric, and narrow on purpose.

## Goals

- keep the benchmark JSON report as the canonical machine-readable artifact
- generate a benchmark-owned self-contained HTML report under `evals/reports/`
- add a CLI path that can open the report in the browser
- make the default experience a cross-entry comparison table
- support drilling into one selected benchmark entry on the same page
- show run summary, failures, and slowest cases for the selected entry
- keep the UI generic enough to work for benchmark entries that compare models,
  prompts, or other entry types
- use `shadcn` components as the UI foundation

## Non-goals

- moving this UI into the main app frontend
- adding successful-case inspection in this item
- expanding the persisted JSON report schema
- adding report history/version browsing
- building a multi-page benchmark UI
- silently degrading through unknown report shapes

## Source Of Truth

- canonical machine-readable report:
  - `evals/reports/<benchmark_id>.json`
- canonical browser-readable report:
  - `evals/reports/<benchmark_id>.html`

The HTML report is a presentation artifact derived from the persisted JSON
report. The JSON report remains the source artifact for machine use.

## UI Contract

### CLI behavior

Existing behavior must remain:

- `benchmark report <benchmark_id>` prints the terminal report
- `benchmark report <benchmark_id> --json` prints JSON

New behavior:

- `benchmark report <benchmark_id> --html`
  - ensures the JSON report exists and is current
  - writes a self-contained HTML file to
    `evals/reports/<benchmark_id>.html`
  - returns the HTML path or otherwise clearly reports where the file was
    written
- `benchmark report <benchmark_id> --open`
  - ensures the JSON report exists and is current
  - ensures the HTML report exists and is current
  - opens the HTML report in the default browser

`--open` is the user-facing path. `--html` exists so the HTML artifact can be
generated in tests, scripts, and review workflows without opening a browser.

### Page structure

The HTML report is a single self-contained page.

Top section:

- benchmark header card
- benchmark ID
- headline metric name
- hosted dataset
- stale/current status
- secondary benchmark metadata only if useful

Main section:

- one comparison table row per benchmark entry
- generic columns, not model-specific wording
- default columns:
  - `label`
  - `status`
  - `headline metric`
  - `completed`
  - `failures`
  - `avg duration`
  - `max duration`
  - `selected timestamp`

Detail section:

- lives on the same page
- updates when a table row is selected
- shows:
  - selected run summary
  - failures section
  - slowest cases section

### Interaction model

- clicking a comparison-table row selects one benchmark entry
- the selected row is visibly highlighted
- the detail pane updates in place
- no router and no separate detail page in this item
- no search or filtering in v1 unless the page becomes unreadable without it

## Rendering Contract

### Technology choice

- keep the renderer under `evals`, not under `frontend/src/App.tsx`
- use a small React-based renderer because the repo already has React tooling
- use `shadcn` as the component system

This repo already has a `shadcn` setup in `frontend/components.json`, existing
`ui` primitives, and the expected supporting packages. That setup should be
reused rather than creating a second design system.

The implementation may use any appropriate `shadcn` components, not just the
few already present in the repo, as long as the resulting UI remains clean and
small.

### Artifact shape

- the generated HTML must be self-contained
- CSS and JS should be embedded so the report is portable as one file
- the HTML renderer reads the persisted JSON report and formats it for human
  inspection

### Failure policy

The renderer should fail cleanly, not opaquely.

If the persisted JSON report does not match the required shape:

- the HTML page should render a clear failure panel
- the page should identify the report file being rendered
- the page should list the missing or invalid required fields when practical
- the UI should not silently degrade into a misleading partial view

This item prefers a clear visible failure state over a broad future-proof
fallback system.

## Phase 1: Static HTML Report Generation

### Objective

Generate a benchmark-owned self-contained HTML report from the persisted JSON
report and keep it under `evals/reports/`.

### Behavioral delta

Before this phase, the benchmark report can be persisted only as JSON and viewed
either in raw form or through terminal text. After this phase, the same
benchmark also has a self-contained HTML report artifact that renders the JSON
report in a human-friendly way.

### Non-goals

- opening the browser in this phase
- successful-case inspection
- JSON report schema changes
- multi-page HTML outputs

### Acceptance gates

1. `benchmark report todo_extraction_bench_v1 --html` creates
   `evals/reports/todo_extraction_bench_v1.html`.
2. The generated HTML is self-contained and, when opened in a browser, renders
   the persisted benchmark report cleanly or shows a clear failure state for an
   unsupported report JSON shape.

### Supporting verification

- renderer-focused unit tests
- CLI tests for `--html`
- browser validation that the page shows:
  - benchmark header
  - cross-entry comparison table
  - same-page selected-entry detail
  - run summary, failures, and slowest cases for the selected entry
- browser validation of the unsupported-report failure state
- browser validation against a generated HTML report

### Phase boundary rule

Do not add browser-opening behavior until static HTML generation passes these
gates.

## Phase 2: Browser Opening And CLI Workflow

### Objective

Make the report UI accessible directly from the benchmark CLI via `--open`.

### Behavioral delta

Before this phase, operators must manually locate and open the generated HTML
file. After this phase, the CLI can generate or refresh the HTML report and
open it directly in the default browser.

### Non-goals

- adding a long-running local server
- changing existing `benchmark report` terminal or JSON semantics
- adding report filtering or search in this item

### Acceptance gates

1. `benchmark report todo_extraction_bench_v1 --open` ensures the JSON and HTML
   report artifacts exist and are current.
2. `--open` opens the benchmark-owned HTML report in the default browser and
   the opened page reflects the same current benchmark state as the persisted
   JSON report it was generated from.

### Supporting verification

- CLI tests for `--open`
- CLI tests that existing `benchmark report`, `benchmark report --json`, and
  `benchmark report --html` behavior remains clear and non-conflicting
- browser validation against the generated HTML report

### Phase boundary rule

The next item must not begin until the CLI can generate and open the static
benchmark report UI cleanly.

## Phase 3: Case Outcome Semantics And Richer Inspection

### Objective

Refine the report semantics so the benchmark UI distinguishes:

- `passed` cases
- `incorrect` cases
- `incomplete` cases

and lets the operator inspect incorrect and incomplete cases with enough detail
to understand what went wrong.

### Behavioral delta

Before this phase:

- the report table shows `completed` and `failures`
- `failures` mixes all non-completed cases under one generic label
- the detail view collapses every incomplete case to the same placeholder
  summary
- completed-but-incorrect cases are not directly visible in the detail view
- the displayed headline metric can remain `1.0` even when some cases are
  incomplete

After this phase:

- the report distinguishes:
  - `passed`: completed and benchmark-correct
  - `incorrect`: completed but benchmark-wrong
  - `incomplete`: no completed evaluation because of runtime, validation, or
    provider failure
- the primary headline shown in the report becomes:
  - `passed cases / total cases`
- `incomplete` cases are counted in the denominator and therefore lower the
  displayed pass rate
- the detail view contains separate sections for:
  - `Incorrect cases`
  - `Incomplete cases`
- the detail view no longer relies on the generic
  `Case span failed before producing a completed evaluation` placeholder as the
  sole explanation for incomplete cases

### Semantics contract

This phase changes the report terminology and presentation contract.

#### Entry-level counts

The comparison table and run summary should present:

- `passed`
- `incorrect`
- `incomplete`

The old `completed` count is no longer the primary operator-facing count. If it
remains available internally, it is a derived quantity:

- `completed = passed + incorrect`

The old `failures` label should not remain as the primary user-facing label,
because it is ambiguous between:

- a completed but incorrect case
- an incomplete case

#### Headline metric

For this UI, the headline shown to the user should be derived from case
outcomes, not reused directly from the existing evaluator aggregate when that
aggregate excludes incomplete cases.

Displayed headline metric:

- `passed / total`

This phase does not require changing the underlying benchmark evaluator itself.
It requires the report layer to compute and display the operator-facing
headline consistently from case outcomes.

### Detail inspection contract

#### Incorrect cases

The `Incorrect cases` section should show, for each completed-but-incorrect
case:

- case ID
- transcript or other case input
- expected output
- actual output

This section is for semantic misses where the run completed, but the benchmark
assertion failed.

#### Incomplete cases

The `Incomplete cases` section should show, for each incomplete case:

- case ID
- transcript or other case input
- expected output
- concrete exception summary or failure reason

When available from the trace data, this section should also show the latest
useful response context, such as:

- last available structured output
- or last available model response excerpt
- or validator feedback that explains the retry/failure path

This phase should not show chain-of-thought as a default inspection surface.
If the trace contains only reasoning text and no valid structured output, the
UI should prefer a concise failure summary and a clear note that no valid
structured result was emitted.

### UI contract changes

#### Comparison table

Default columns should be updated to favor the new outcome taxonomy:

- `label`
- `status`
- `headline`
- `passed`
- `incorrect`
- `incomplete`
- `avg duration`
- `max duration`
- `selected timestamp`

#### Detail view

The selected-entry detail should contain:

- selected run summary
- `Incorrect cases`
- `Incomplete cases`
- `Slowest cases`

The `Incorrect cases` and `Incomplete cases` sections may use cards, tables, or
another compact list format, as long as they remain readable and clearly
separated.

### Non-goals

- changing the benchmark evaluator contract itself in this phase
- introducing full successful-case inspection for all cases
- exposing raw chain-of-thought by default
- turning the report into a full trace browser

### Acceptance gates

1. In the benchmark report UI, each entry clearly distinguishes `passed`,
   `incorrect`, and `incomplete` cases, and the displayed headline metric is
   `passed / total` for that entry.
2. In the selected-entry detail view, operators can inspect both:
   - at least one incorrect completed case with input, expected output, and
     actual output
   - at least one incomplete case with input, expected output, and a concrete
     failure reason rather than the generic placeholder summary

### Supporting verification

- report-layer tests for outcome classification:
  - passed
  - incorrect
  - incomplete
- tests proving the displayed headline is derived from `passed / total`
- tests proving incomplete cases no longer collapse to the generic placeholder
  as their only explanation
- browser validation on a benchmark entry with incorrect completed cases
- browser validation on a benchmark entry with incomplete cases

### Phase boundary rule

The report should not be considered an operator-usable benchmark inspector
until incorrect and incomplete cases are separated and inspectable under this
contract.
