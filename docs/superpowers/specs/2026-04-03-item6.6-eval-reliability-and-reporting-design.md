# Item 6.6 Design: Eval Reliability And Reporting

Scope: harden the extraction eval tracks so transient provider failures are handled more deliberately, failure-aware reporting is explicit, and detailed per-experiment analysis can be merged back into the local experiment artifacts from Logfire after each run.

## Why this exists

Items 6 and 6.5 gave us two working eval tracks:

- transcript-only extraction evals on finalized transcript text
- incremental replay evals where each model carries forward its own prior todos

The first real runs surfaced two follow-up needs:

- reliability gaps such as provider-side `503` failures are currently mixed into the same experience as semantic model failures
- the reporting story is split between compact local artifacts and richer Logfire traces, but that split is not explicit enough yet

Today the local artifacts are good for reproducibility and quick inspection, while Logfire is better for timing, token, cost, payload, and failure debugging. Item 6.6 exists to make that boundary intentional and easier to use.

## Goals

- Add an explicit retry policy for transient provider or task failures
- Make failure-aware run accounting visible in local results
- Distinguish provider transport failures from output-validation and evaluator failures
- Keep local artifacts understandable and reproducible
- Use Logfire as the detailed source of truth for per-case timing, token, cost, payload, and failure diagnostics
- Merge Logfire-derived timing and token data into each experiment artifact so comparisons do not require repeated interactive queries

## Non-goals

- Changing the Item 6 or 6.5 dataset contracts
- Expanding v1 scoring beyond count-based evaluators
- Replacing Logfire as the detailed observability layer
- Making eval success depend on Logfire availability
- Building a general-purpose analytics UI beyond the existing artifacts and Logfire views

## Approved Design Decisions

- Base eval artifacts are written first, then enriched in place after the run
- Local experiment artifacts become the source of truth for experiment identity, run accounting, and per-case comparisons
- Logfire remains the source of truth for detailed per-case timing, token, cost, payload, and deep failure inspection
- A post-run enrichment step queries Logfire and merges richer timing or token detail back into each experiment artifact
- Logfire enrichment is best-effort and must not cause the eval run itself to fail
- Retry policy is added at the eval-runner level for transient task failures rather than by overloading `repeat`

## Reliability Model

### Repeat vs retry

The existing `--repeat` flag keeps its current meaning:

- run each case multiple times to sample repeatability
- do not treat those repeated runs as recovery attempts for failures

For artifact accounting, repeats should be represented as separate executed case attempts rather than collapsed into one logical row. If a dataset has 9 cases and `--repeat 3`, the experiment artifact should represent 27 case attempts, each with its own outcome and repeat ordinal.

Item 6.6 adds a separate retry policy for transient failures. This policy should be configurable per run and should apply only to failures that are likely temporary, such as provider transport errors or short-lived upstream unavailability.

### Retry scope

The retry policy should apply at the eval-runner task level rather than by changing the semantic meaning of the model prompt or the dataset. The runner should be able to retry a failed case when the failure matches a transient category.

The first version should focus on:

- provider HTTP failures such as `503`
- connection and timeout style request failures

The first version should not automatically retry:

- deterministic evaluator failures
- output-validation failures caused by schema mismatch
- semantic score failures such as count mismatch

### Retry reporting

Retry behavior should be visible in the enriched experiment artifact so we can answer:

- how many retries were attempted
- which cases recovered after retry
- which cases still failed after exhausting retries

The base artifact should already record retry-attempt counts and recovered-after-retry state per executed case attempt so that this information remains available even when enrichment is skipped.

## Reporting Source Of Truth

### Local experiment artifact

The local experiment artifact should stay durable and easy to inspect. Before enrichment it should continue to record:

- timestamp
- dataset name
- experiment id
- model, provider, and thinking-mode metadata for that single experiment run
- prompt family, version, and sha
- git branch and commit sha
- repeat count, max concurrency, and retry policy used
- trace id and span id
- enrichment status fields such as `enrichment_status` and `enrichment_reason`, initialized on first write
- one row per executed case attempt with:
  - case name
  - repeat ordinal when `--repeat > 1`
  - expected value
  - predicted value when available
  - status
  - retry-attempt count
  - recovered-after-retry state
  - failure category and exception summary when a case does not complete
  - trace or span ids

Item 6.6 should add explicit top-level accounting fields such as:

- `completed_cases`
- `failure_count`
- `overall_case_success_rate`
- failure counts by category when classification is available

After enrichment, that same file should also record the main Logfire-derived comparison fields for that experiment:

- experiment-level timing stats such as average, minimum, maximum, and a high-percentile latency
- per-case timing
- per-case token and cost data when available
- deeper per-case exception summaries when available from Logfire

This gives us one trustworthy local file per experiment even when a UI only shows assertion pass rate on completed cases.

### Logfire detailed data

Logfire should remain the detailed source for:

- per-case timing and timing distributions
- per-case token and cost data
- full case inputs, outputs, scores, and assertions
- exception type and exception message
- nested agent and provider spans needed for debugging

This matches what the current instrumentation already captures well and avoids duplicating a large amount of observability data into repo artifacts.

## Post-run Experiment Artifact Enrichment

After an eval run completes, the harness should optionally perform a Logfire enrichment step keyed from the experiment trace and metadata. This step downloads the main comparison metrics for that experiment and merges them into the existing experiment artifact.

### Enriched experiment artifact contents

The enriched experiment artifact should include enough detail for common comparison work without needing fresh Logfire queries:

- experiment-level `completed_cases`, `failure_count`, and `overall_case_success_rate`
- failure counts by category
- experiment-level timing stats such as average, minimum, maximum, and a high-percentile latency
- one row per executed case attempt with:
  - case name
  - repeat ordinal
  - expected value
  - predicted value
  - status
  - duration
  - retry-attempt count
  - recovered-after-retry state
  - failure category if failed
  - exception summary if failed
  - token and cost metrics when available

The enriched artifact should also preserve:

- experiment trace id
- per-case trace and span ids

This keeps the experiment artifact useful on its own while still making it easy to jump back into Logfire when deeper inspection is needed.

### Enrichment behavior

The enrichment step should:

- use normal Logfire query access, not depend on Codex MCP
- retry or poll briefly for late-arriving telemetry
- degrade gracefully if Logfire credentials are missing or the query fails
- keep the experiment artifact valid and mark enrichment status there rather than failing the whole eval run
- preserve the base case rows and outcome accounting even when enrichment is skipped or partial

## Failure Classification

Item 6.6 should standardize failure categories so reporting is easier to reason about.

The first useful categories are:

- `provider_transport_failure`
- `output_validation_failure`
- `evaluator_failure`
- `unexpected_task_failure`

Classification should be derived from the exception type and run context, and it should appear in both the local base artifact and the enriched experiment artifact when possible.
Classification should be derived from the exception type and run context, and it should appear in the experiment artifact both before and after enrichment when possible.

## Relationship To Items 6 And 6.5

- Item 6 remains the transcript-only extraction eval track
- Item 6.5 remains the incremental replay extraction eval track
- Item 6.6 is shared hardening and reporting infrastructure for both tracks
- Item 6.6 does not change the dataset boundaries or the v1 count-based scoring contract of either track

## Success Criteria

Item 6.6 is successful when:

- a provider-side transient failure can be retried deliberately without using `--repeat`
- local artifacts make completed-case pass rate versus overall run success unambiguous
- timing and token or cost analysis can be reviewed locally from each enriched experiment artifact
- failure classes make it obvious whether a bad run was caused by provider transport, output validation, evaluator logic, or something unexpected
- the eval harness still works when Logfire enrichment is unavailable, with a clear degraded outcome instead of a hard failure

## Open Questions After The First 6.6 Pass

- Should the retry policy be global across both eval tracks or configurable per experiment?
- How long should the enrichment step wait for Logfire ingestion before declaring the artifact enrichment partial?
- Which percentile latency is most useful for comparison in practice: p90, p95, or both?

## Amendment After First Verification Run

The first implementation pass validated the retry and Logfire query mechanics, but it also showed that a run-level `summary.json` creates too much duplication and hides the most useful comparison surface behind an extra layer.

The amended reporting direction for Item 6.6 is:

- do not keep a run-level `summary.json` as the primary local reporting artifact
- enrich each experiment artifact in place after the run
- treat one experiment file as the only local file needed to inspect that provider or model configuration run
- keep case-level timing in that experiment file so case-by-case latency comparisons remain possible
- avoid duplicated case lists across a run summary and per-experiment files

If a lightweight run index is ever reintroduced later, it should stay strictly top-level and should not duplicate case rows that already live in the experiment artifacts.
