# Item 6.9 Design: Benchmark Abstraction For Eval Orchestration

Scope: define a repo-level benchmark abstraction that sits above Pydantic experiments, uses Logfire as the canonical experiment-results store, and supports additive comparison workflows without redefining Pydantic's own terms.

## Why this exists

Once Item 6.8 aligns the base lifecycle with Pydantic Evals and Logfire, the repo still needs one more concept above a single experiment.

That concept is needed to answer questions like:

- "compare these five models on the same extraction task"
- "rerun only the missing prompt variants"
- "add two new models later without rerunning the original baseline"
- "compare a prompt sweep for one fixed model"
- "run a deliberate model x prompt grid and then analyze it as one coherent evaluation effort"

Pydantic gives us dataset, experiment, and evaluation report. It does not give us a repo-level object that defines a family of comparable experiments and the orchestration rules for filling that family over time.

That is what `benchmark` should mean in this repo.

This item also chooses a strict ownership boundary:

- experiments remain benchmark-agnostic
- benchmarks reference experiments
- compatibility with a benchmark contract is validated on the benchmark side, not stored as experiment provenance

This item also relies on one identity clarification:

- `experiment_id` remains the stable identity for one evaluated configuration
- `experiment_run_id` is the unique identity for one concrete experiment execution
- benchmark membership should point to `experiment_run_id`, not to the stable `experiment_id`

## Goals

- define `benchmark` as a term that does not fight Pydantic terminology
- support single-axis and multi-axis comparison work
- make the compatibility contract explicit instead of inferring it from directory layout
- allow additive population of missing experiments across multiple batch executions
- use Logfire metadata to discover existing experiments and compare results
- keep benchmark definitions in git so they are reviewable and reproducible
- keep benchmark membership stable unless the benchmark object is explicitly edited

## Non-goals

- replacing Pydantic's `Dataset`, `Experiment`, or `EvaluationReport` concepts
- making benchmark a Logfire-native first-class object
- implementing the Item 6.8 lifecycle migration itself
- building a full benchmark UI in this item
- supporting arbitrary unmanaged ad hoc comparisons with no manifest or metadata contract
- storing benchmark membership as experiment-side metadata

## Core definition

`Benchmark` is a git-tracked object that defines a family of comparable experiments and explicitly lists which experiments belong to it.

A benchmark does not store canonical experiment results itself. Instead, it defines:

- what objective is being measured
- which dataset and evaluator contract must stay fixed
- which configuration fields are fixed
- which configuration fields are allowed to vary
- how to identify the expected experiment coordinates
- which experiment run IDs are attached to the benchmark
- how to validate attached experiments against the benchmark contract

## Relationship to Pydantic and Logfire

- Pydantic `Dataset` remains the static eval contract
- Pydantic `Experiment` remains one execution of that dataset
- Pydantic `EvaluationReport` remains the result of that one execution
- Logfire remains the canonical store for experiment results
- `Benchmark` is the repo-level orchestration and comparison layer above those concepts

## Design decisions

### 1. Benchmarks are manifest-based, not implicit query bundles

There are two broad ways to define a benchmark:

- implicit query-only benchmarking, where the benchmark is reconstructed later from whatever metadata happens to exist
- explicit manifest-based benchmarking, where the benchmark contract is written and versioned in git

Choose the manifest-based design.

It is more work up front, but it is much safer because:

- the intended comparison contract is reviewed before running expensive experiments
- fixed versus varying fields are explicit
- additive later runs can validate against the same contract
- future agents do not need to guess which past experiments were supposed to be comparable

### 2. A benchmark owns the comparison contract

Each benchmark manifest should define:

- `benchmark_id`
- `title`
- `description`
- `suite`
- `dataset_name`
- `dataset_version`
- `dataset_sha`
- `evaluator_contract_version`
- `evaluator_contract_sha`
- `fixed_config`
- `axes`
- `attached_experiment_runs`
- `baseline_selector` when relevant

The benchmark contract is what determines whether two experiments are compatible enough to compare inside that benchmark.

Under this design, the benchmark also owns membership.

That means:

- the benchmark stores which `experiment_run_id`s are attached to it
- experiments do not store `benchmark_id`
- the same experiment run may be attached to more than one benchmark
- compatibility is checked when reporting, not enforced by mutating experiment records

### 3. A benchmark coordinate identifies one planned experiment

Each experiment that belongs to a benchmark should have a coordinate.

A coordinate is the full assignment of benchmark axis values for that experiment.

Examples:

- model benchmark coordinate: `{"model_name": "gemini-2.5-flash"}`
- prompt benchmark coordinate: `{"prompt_sha": "abc123"}`
- grid benchmark coordinate: `{"model_name": "gemini-2.5-flash", "prompt_sha": "abc123"}`

One coordinate maps to one planned experiment under one benchmark contract.

Coordinates are part of the benchmark object, not part of experiment membership metadata.

They are used to:

- define the intended comparison surface
- determine which experiments should be launched when running from a benchmark
- explain coverage gaps in reports

They are not, by themselves, membership. An experiment run becomes part of a benchmark only when its `experiment_run_id` is attached to that benchmark.

### 4. Experiment configuration identity and experiment run identity stay separate

This benchmark design depends on keeping stable configuration identity separate from unique run identity.

Use these meanings:

- `experiment_id`: stable identity for one evaluated configuration
- `experiment_run_id`: unique identity for one concrete execution of that configuration
- `batch_id`: grouping identity for one CLI launch event that may produce several experiment runs

Why this matters:

- repeated reruns of one config should usually keep the same `experiment_id`
- benchmark membership must point to one concrete run, not to a reusable config label
- `experiment_run_id` gives the benchmark a single unambiguous attachment target

This avoids overloading `experiment_id` with two incompatible jobs.

### 5. Single-axis should be the default, multi-axis should be explicit

Most benchmarks should vary one primary axis.

Examples:

- compare models while prompt stays fixed
- compare prompts while model stays fixed

Multi-axis benchmarks are allowed, but they should be deliberate because they increase experiment count quickly and make analysis harder.

Recommended rule:

- default benchmark shape: one axis
- multi-axis benchmark: allowed only when the manifest explicitly declares more than one axis

### 6. Benchmarks are populated across batch executions

One benchmark can be filled over time through multiple batch executions.

Example:

- batch A runs the original five models
- batch B later adds two DeepInfra models
- the benchmark object attaches the five resulting `experiment_run_id`s from batch A
- later it attaches the two resulting `experiment_run_id`s from batch B

This preserves provenance because each experiment still belongs to its own batch execution and its own experiment record.

### 7. Benchmark membership is explicit, not inferred from experiment metadata or filesystem layout

An experiment run belongs to a benchmark when its `experiment_run_id` appears in the benchmark's membership list.

This keeps benchmark boundaries stable:

- attached experiments do not silently appear because they happen to match a query
- reports do not silently widen when new compatible experiments land in Logfire
- benchmark membership changes only when someone edits the benchmark object

Compatibility still matters, but it is handled on the benchmark side.

At report time, each attached experiment should be classified as:

- attached and compatible
- attached but incompatible
- attached but missing from Logfire

The report should compare only the compatible attached experiments by default while still surfacing incompatible or missing attachments clearly.

### 8. Benchmark reporting is derived, not native

There is no need to pretend that Pydantic already has a native "benchmark report" object.

Instead:

- a benchmark report is a derived view over many experiments
- the source data comes from Logfire
- local tooling may export markdown, JSON, or CSV summaries for convenience
- the report is built from the benchmark's attached `experiment_run_id`s, not from an open-ended membership query

This keeps the concept honest:

- Pydantic report: one experiment
- benchmark report: repo-defined aggregation across many experiments

## Benchmark manifest shape

The manifest should be compact but explicit.

Required sections:

- benchmark identity
- objective
- dataset contract
- evaluator contract
- fixed configuration
- axes
- attached experiment membership
- compatibility rules
- reporting defaults

Suggested axis shape:

- `name`
- `field`
- `values`
- `description`

Suggested compatibility rules:

- hard match:
  - suite
  - dataset name
  - dataset version
  - dataset sha
  - evaluator contract version
  - evaluator contract sha
- soft warnings:
  - git commit sha
  - max concurrency
  - region
  - infrastructure details

Suggested membership section:

- `attached_experiment_runs`
- required `experiment_run_id`
- optional per-experiment note or label
- optional coordinate-to-experiment mapping aid for operator readability

The benchmark object should stay small and readable in git. It should not duplicate canonical experiment result data that already lives in Logfire.

## Benchmark lifecycle

### 1. Author benchmark manifest

The author defines the benchmark in git with explicit fixed fields, axes, and an empty or partially populated membership list.

### 2. Resolve target coordinates

Tooling expands the axis values into the expected experiment coordinates.

For a one-axis benchmark, this is a simple list.

For a multi-axis benchmark, this is the cartesian product of the axis values.

### 3. Launch from benchmark or attach existing experiments

There are two supported ways to populate a benchmark.

#### Launch from benchmark

Tooling resolves the requested coordinates, launches the corresponding experiments through the Item 6.8 lifecycle, collects the resulting `experiment_run_id`s, and appends them to the benchmark membership list.

#### Attach existing experiments

An operator may add existing `experiment_run_id`s to the benchmark membership list after the fact.

This supports a "formalize later" workflow:

- run exploratory experiments first
- define the benchmark later
- attach the experiment run IDs that should belong to that benchmark

The attachment step is trust-by-default. It does not need to reject mismatches up front. Contract validation happens during reporting.

### 4. Determine coverage and gaps

Once a benchmark has expected coordinates and attached experiments, tooling can compare the two to show:

- attached coordinates with compatible experiments
- attached coordinates with incompatible experiments
- expected coordinates that are still missing
- extra attached experiments that do not map cleanly to the intended benchmark shape

This supports "run missing" workflows without requiring experiment-side benchmark metadata.

### 5. Generate derived comparison output

Comparison and summary views are built from Logfire experiment results.

Possible outputs:

- interactive comparison in Logfire
- local markdown summary
- local JSON export
- local CSV table

The exported summary is derived output, not the canonical store.

By default, the report should be stable:

- it reads the benchmark's attached experiment run IDs
- it validates those experiments against the benchmark contract
- it compares only the compatible attached experiments

This means benchmark reports change when:

- the benchmark membership list changes
- attached experiments in Logfire change visibility or availability
- the benchmark contract itself is edited

They do not change merely because a new compatible experiment happened to land in Logfire.

## Expected behavior changes for benchmark authors and operators

This item changes how an eval operator organizes comparison work above individual experiments.

The primary user for this section is:

- the benchmark author defining benchmark manifests
- the eval operator launching experiments, attaching experiments, and generating reports

The behavior changes below are intentionally written in operator terms so they can drive implementation planning, validation steps, and acceptance tests.

These behavior changes are normative.

That means:

- implementation work should be decomposed by these behaviors
- automated tests should be derived from these behaviors and their acceptance criteria
- the item should not be considered complete until those behavior-level automated tests exist and pass
- manual checks may supplement this coverage, but they do not replace it

### 1. Create a benchmark without launching experiments

#### Expected behavior

An operator can define a benchmark in git before any experiments are launched for it.

Creating a benchmark should let the operator declare:

- the benchmark identity and purpose
- the fixed contract fields
- the allowed axes
- the expected comparison surface
- an empty or partially populated membership list

Creating the benchmark does not require launching experiments immediately.

#### Boundaries

- creating a benchmark does not create experiments
- creating a benchmark does not modify Logfire state
- creating a benchmark does not imply that all expected coordinates already exist

#### Acceptance criteria

- given a benchmark manifest with valid identity, contract, and axis fields
- when the operator creates the benchmark with no attached experiments
- then the benchmark is a valid object with an empty membership list
- and the benchmark can still resolve expected coordinates
- and the benchmark can still be reported later as empty or incomplete without error

### 2. Launch experiments from a benchmark

#### Expected behavior

An operator can use a benchmark to launch experiments for some or all expected coordinates.

When launched from a benchmark, tooling should:

- resolve the requested coordinates
- launch the corresponding experiments through the Item 6.8 lifecycle
- collect the resulting `experiment_run_id`s
- append those `experiment_run_id`s to the benchmark membership list

This should work whether the launch covers:

- all expected coordinates
- only missing coordinates
- an explicit subset chosen by the operator

#### Boundaries

- launching from a benchmark does not make experiments store benchmark membership
- benchmark membership is recorded only on the benchmark side
- different launched experiments may still belong to different `batch_id`s depending on how the operator runs them

#### Acceptance criteria

- given a benchmark with expected coordinates and no attached experiments
- when the operator launches a subset of benchmark coordinates
- then the corresponding experiments are executed through the normal experiment lifecycle
- and each successful launched experiment yields an `experiment_run_id`
- and those `experiment_run_id`s are attached to the benchmark
- and the benchmark membership list reflects exactly the launched successful experiments
- and the attached experiments remain queryable in Logfire as ordinary experiments

### 3. Attach existing experiments to a benchmark

#### Expected behavior

An operator can define a benchmark after experiments already exist and attach those existing `experiment_run_id`s to the benchmark.

This supports a "formalize later" workflow:

- run experiments first
- create the benchmark later
- attach the experiment runs that should belong to that benchmark

Attachment is trust-by-default.

That means the attach step should not need to reject experiments immediately because they may later turn out to be incompatible. Compatibility is checked during reporting.

#### Boundaries

- attaching an experiment does not mutate the experiment record in Logfire
- attaching an experiment does not write benchmark membership onto the experiment
- attaching an incompatible experiment is allowed, but it must be surfaced during reporting

#### Acceptance criteria

- given an existing `experiment_run_id` already recorded in Logfire
- and a benchmark whose membership list does not yet include that experiment
- when the operator attaches that `experiment_run_id` to the benchmark
- then the benchmark membership list includes the experiment
- and the underlying experiment record in Logfire is unchanged
- and a later report classifies the attached experiment run as compatible or incompatible against the benchmark contract

### 4. Inspect benchmark coverage and gaps

#### Expected behavior

An operator can inspect how the benchmark's attached experiments relate to the intended benchmark shape.

Coverage inspection should show:

- which expected coordinates have attached compatible experiments
- which expected coordinates are still missing
- which attached experiments are incompatible with the benchmark contract
- which attached experiments do not map cleanly to the intended coordinate layout

This gives the operator a concrete basis for deciding what to launch next.

#### Boundaries

- coverage inspection should not silently attach new experiments
- coverage inspection should not redefine the benchmark contract
- coverage inspection should distinguish membership problems from compatibility problems

#### Acceptance criteria

- given a benchmark with expected coordinates and a mixed membership list
- where some attached experiments are compatible, some are incompatible, and some expected coordinates are missing
- when the operator inspects benchmark coverage
- then the output distinguishes compatible coverage, incompatible attachments, and missing coordinates
- and the operator can identify which coordinates remain to be filled
- and no new benchmark members are added as a side effect of inspection

### 5. Generate a benchmark report from attached experiments

#### Expected behavior

An operator can generate a benchmark report that uses only the benchmark's attached experiment runs as its candidate input set.

At report time, the system should:

- fetch attached experiment runs from Logfire
- validate each attached experiment against the benchmark contract
- compare only the compatible attached experiments by default
- surface incompatible or missing attached experiments clearly

The report should be stable. It should not silently widen because unrelated but compatible experiments later appeared in Logfire.

#### Boundaries

- the report is derived output, not a canonical stored benchmark result
- report generation does not change benchmark membership by itself
- report generation does not infer new benchmark members from open-ended Logfire queries

#### Acceptance criteria

- given a benchmark with attached experiment run IDs
- when the operator generates a benchmark report
- then the report fetches those attached experiment runs from Logfire
- and the report validates each attachment against the benchmark contract
- and the default comparison surface includes only compatible attached experiments
- and incompatible or missing attached experiments are called out separately
- and rerunning the report without changing the benchmark membership or contract does not add new experiments merely because compatible runs appeared in Logfire

### 6. Update a benchmark across multiple batches

#### Expected behavior

An operator can grow one benchmark over time by attaching experiments produced by separate batch executions.

This should support workflows like:

- launch five experiments in batch A
- later launch two more in batch B
- attach all seven experiments to one benchmark
- report on the benchmark as one curated comparison set

The benchmark should outlive any single batch execution.

#### Boundaries

- batch provenance remains on the experiment side through `batch_id`
- benchmark membership remains on the benchmark side
- one benchmark may contain experiments from several different batches

#### Acceptance criteria

- given a benchmark with experiments attached from one earlier batch
- when the operator later attaches experiments produced by a different batch
- then the benchmark membership list contains experiments from both batches
- and coverage/reporting treat them as members of one benchmark
- and batch provenance remains discoverable from the attached experiments themselves
- and the benchmark object remains the only place where benchmark membership is defined

## Testing and validation requirements

The testing strategy and validation strategy for Item 6.9 should be driven directly by the expected behavior changes above.

This is a prescriptive requirement, not a suggestion.

### 1. Behavior-driven test planning is required

Implementation planning should break the work down by operator-visible behavior, not only by files or helper functions.

At minimum, the implementation plan should map each behavior section above to one or more automated tests:

- create a benchmark without launching experiments
- launch experiments from a benchmark
- attach existing experiments to a benchmark
- inspect benchmark coverage and gaps
- generate a benchmark report from attached experiments
- update a benchmark across multiple batches

### 2. Automated tests are required before the item is considered done

Before Item 6.9 can be considered complete:

- each behavior change above must have automated test coverage
- each behavior's acceptance criteria must be represented in those automated tests
- the automated test suite for this item must pass

Manual validation is useful for confidence, but it is not sufficient on its own.

### 3. Manual validation supplements automated coverage

Manual or operator-level validation may still be useful for:

- confirming CLI ergonomics
- confirming benchmark manifests are readable and usable
- confirming end-to-end behavior against a real Logfire-backed environment

But those checks are secondary.

They should confirm the automated coverage, not substitute for it.

### 4. Recommended validation layers

The implementation should validate these behaviors at more than one level.

Recommended layers:

- unit tests for benchmark manifest parsing, membership updates, and compatibility classification
- integration tests for benchmark launch, attach, coverage, and report flows against a controlled Logfire-facing test surface
- operator acceptance checks for one or two realistic benchmark workflows

The exact file layout can change during implementation, but the behavior coverage should remain anchored to the sections above.

### 5. Definition of done for this item

This item is not done when:

- the benchmark data structures exist
- the CLI commands exist
- a manual happy path appears to work once

This item is done only when:

- the benchmark behavior model described in this spec is implemented
- the behavior-level automated tests derived from this spec exist
- those automated tests pass
- any required manual validation is recorded as supplementary evidence

## Relationship to batch execution

`Batch execution` remains the right term for one CLI invocation.

A batch execution:

- may be launched from a benchmark or outside any benchmark workflow
- may cover all benchmark coordinates or only a subset
- should always produce one `batch_id`

A benchmark is longer-lived than a batch execution. It is normal for one benchmark to be populated over several separate batches.

Under this design:

- `batch_id` remains experiment provenance
- benchmark membership remains a property of the benchmark object
- a benchmark may attach experiments that came from several different batches

## Open questions for implementation planning

- what file format should benchmark manifests use in this repo
- how much of benchmark launch and attachment should live in the existing runners versus a separate benchmark CLI
- what benchmark summary outputs are worth generating locally once Logfire is canonical
- whether benchmark baselines should be expressed as one coordinate, a subset of coordinates, or simply as the first compatible attached state
- what operator experience should exist for attaching historical `experiment_run_id`s to a benchmark
- whether the membership list should support lightweight annotations per experiment

## Success criteria

This item is successful when:

- the repo has a clear term above experiment that does not overload Pydantic language
- a benchmark can describe both fixed contract fields and allowed varying axes
- additive later runs can populate one benchmark across multiple batches
- a benchmark can be defined before or after experiments exist
- experiments remain benchmark-agnostic
- benchmark membership is explicit and stable
- benchmark reports validate attached experiments against the contract rather than inferring membership from filesystem layout or experiment-side benchmark metadata
- a future implementation can query which benchmark experiments already exist and run only the missing ones
- the implementation plan and automated tests are organized around the operator-visible behavior changes defined in this spec
- the item is not considered complete without passing automated coverage for those behavior changes
