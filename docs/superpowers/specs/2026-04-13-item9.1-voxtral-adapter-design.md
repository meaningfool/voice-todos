# Item 9.1 Design: Single-Stream Voxtral Adapter

Scope: implement a single-stream Mistral Voxtral adapter behind the Item 8 STT
session seam, using the realtime semantics confirmed by the Item 9 spike.

## Why this exists

Item 9 proved enough about Voxtral's realtime behavior to move from discovery to
implementation:

- the session lifecycle is viable for a second provider
- realtime transcript updates arrive as additive `transcription.text.delta`
  events
- the terminal stop barrier is `transcription.done`
- `TranscriptionStreamDone.text` matched the accumulated streaming text in the
  captured traces
- no Soniox-style endpoint or per-token finality signals were observed

That means we can now write a focused integration spec for the adapter itself,
without pretending Voxtral supports Soniox transcript semantics.

## Goals

- add a Voxtral-backed `SttSession` implementation in `backend/app/`
- make provider selection support `mistral` alongside the existing Soniox path
- map Voxtral realtime events into the app-level STT contract in a way that is
  honest about missing Soniox-style semantics
- make the stop path wait for Voxtral's terminal transcript event
- keep the browser `/ws` contract stable enough for the existing frontend to
  remain functional

## Non-goals

- dual-delay
- delay tuning or benchmark work
- redesigning the transcript UI
- designing a new incremental extraction strategy
- changing the default provider away from Soniox

## Inputs From Item 9

The adapter design should treat these as current evidence-backed facts:

- observed event sequence:
  - `session.created`
  - `session.updated`
  - additive `transcription.text.delta`
  - terminal `transcription.done`
- no `transcription.segment` events observed in the captured fixtures
- no observed endpoint marker
- no observed per-token finality
- `transcription.done.text` matched accumulated delta text in the captured runs

Reference:

- `docs/references/2026-04-13-item9-voxtral-realtime-spike-findings.md`

## Design Decisions

### 1. Add a dedicated `stt_mistral.py` adapter module

Follow the Item 8 shape:

```text
backend/app/
  stt.py
  stt_factory.py
  stt_soniox.py
  stt_mistral.py
```

Responsibilities of `stt_mistral.py`:

- create the realtime Voxtral session
- stream browser audio to Mistral
- translate provider-native events into `SttEvent`
- implement provider-specific stop behavior

`ws.py` must not learn Mistral SDK details directly.

### 2. Preserve the app-level session contract, but not fake Soniox semantics

The adapter should continue implementing the existing `SttSession` protocol:

- `send_audio(...)`
- `request_final_transcript()`
- `end_stream()`
- `wait_for_final_transcript()`
- `close()`
- async iteration over `SttEvent`

But the Voxtral adapter should not invent false Soniox-like meaning.

Specifically:

- translated transcript tokens should be marked `is_final=True`
- `finalization_state` should remain `NOT_OBSERVED` during normal text-delta
  events
- `endpoint_state` should remain `UNSUPPORTED`
- the adapter should surface final transcript readiness through
  `wait_for_final_transcript()` when `transcription.done` arrives

Reason:

- additive deltas are the only transcript behavior actually observed
- we need the adapter to expose the truth of the provider, not an imitation of
  Soniox

### 3. Treat additive deltas as normal transcript text in the first integration

Because no replaceable interim tail was observed, the first Voxtral path should
feed the browser transcript as normal accumulated text.

That implies:

- frontend live transcript remains usable
- the Voxtral path does not participate in Soniox-style gray interim behavior
- the adapter should emit tokens that the current reducer can append as normal
  text

This is an adapter-aligned compatibility choice, not a final UX decision.

### 4. Gate stop handling on `transcription.done`

For Voxtral, the adapter's stop-time final transcript barrier should be the
terminal `transcription.done` event.

Expected adapter behavior:

- `request_final_transcript()` should flush pending audio if needed
- `end_stream()` should close audio input to the provider
- `wait_for_final_transcript()` should resolve when `transcription.done` is
  observed
- the adapter should keep the final transcript text from `done.text` available
  to the app-level stop path

This replaces the Soniox-specific:

- `finalize`
- wait for `<fin>`
- then proceed

sequence.

### 5. Expose the final transcript explicitly to the stop path

Item 9 showed that `transcription.done.text` is the best current candidate for
stop-time transcript authority. The implementation should therefore make the
final transcript available explicitly rather than forcing the app to reconstruct
it only from deltas.

The cleanest approach is:

- extend the app-level `SttSession` contract with a read-only property or method
  for the final transcript text after completion

Candidate shape:

```python
class SttSession(Protocol):
    ...
    @property
    def final_transcript_text(self) -> str | None: ...
```

Why this change is justified:

- Soniox and Voxtral reach "final transcript ready" through different provider
  semantics
- the app-level stop path needs one neutral way to ask for the authoritative
  final transcript, if available
- this keeps provider-specific transcript authority inside the adapter rather
  than in `ws.py`

### 6. Keep unsupported boundary semantics explicit

The Voxtral adapter should report capabilities honestly:

- finalization boundary exposure: `False`
- endpoint boundary exposure: `False`

Translated `SttEvent` values should reflect that:

- `finalization_state=BoundaryState.NOT_OBSERVED` for ordinary text-delta events
- `endpoint_state=BoundaryState.UNSUPPORTED`

This preserves the contract distinction Item 8 introduced:

- boundary not observed
- boundary unsupported

The app can then branch intentionally rather than accidentally assuming Soniox.

### 7. Limit app-level changes to the minimum needed for provider truthfulness

The adapter item should not redesign extraction behavior, but it may require
small app-level changes so Voxtral can fit honestly behind the seam.

Allowed changes:

- `stt.py`: extend the protocol if the final transcript accessor is added
- `stt_factory.py`: support provider selection for `mistral`
- `config.py`: ensure Mistral STT settings are available
- `ws.py`: use the final-transcript accessor during stop if present

Not allowed in this item:

- a new extraction policy
- delay controls exposed to the product
- special frontend rendering modes

## Testing Requirements

At minimum, implementation tests should prove:

- Mistral provider selection works through `stt_factory.py`
- translated Voxtral delta events become additive app transcript tokens
- Voxtral reports unsupported endpoint semantics explicitly
- `wait_for_final_transcript()` resolves on `transcription.done`
- the final transcript accessor exposes `done.text`
- the stop path can use that final transcript without depending on Soniox
  `<fin>` semantics

Live vendor calls are not required for this item's deterministic test suite.
The Item 9 trace fixtures remain the evidence base.

## Success Criteria

This item is complete when:

- `stt_factory.py` can construct a Voxtral session when `stt_provider=mistral`
- the adapter implements the `SttSession` contract cleanly
- Voxtral stop handling waits for `transcription.done`
- the app can obtain the authoritative final transcript from the adapter
- the existing browser `/ws` flow remains operational with Soniox unchanged
- deterministic tests cover the new adapter behavior
