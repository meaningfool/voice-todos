# Item 6.5 Design: Incremental Extraction Replay Evals

Scope: define a separate local LLM eval track that replays ordered extraction snapshots from recorded sessions so each model carries forward its own previously extracted todos and is scored on the final session outcome.

## Why this exists

Item 6 answers the from-scratch extraction question:

- given the same finalized transcript, which LLM setup most reliably extracts the right number of todos at acceptable latency and cost?

Production behavior is different. The extraction path can run multiple times during one session, and each later call receives the current transcript snapshot plus that same model's previously extracted todos. This creates a model-specific trajectory:

- a stronger model can benefit from its own cleaner prior outputs
- a weaker model can carry forward its own hallucinations or over-splits
- the final user-visible result can be wrong partly because earlier intermediate outputs were wrong

We therefore need a second eval question:

- given the same ordered extraction snapshots from a recorded session, which LLM setup most reliably ends with the right final todo count when it must live with its own prior outputs?

This design intentionally keeps the first incremental replay pass narrow so we can measure the carried-state behavior without adding text reconciliation, intermediate ground-truth labeling, or judge-model complexity.

## Goals

- Compare multiple LLM extraction candidates on the same replayed session dataset
- Preserve model-specific carry-forward behavior across extraction steps
- Keep experiments local and repeatable
- Score the final session outcome as the primary metric
- Store per-step outputs for later manual analysis without turning them into scored labels

## Non-goals

- Evaluating STT quality itself
- Replaying raw Soniox event mechanics in the dataset contract
- Encoding trigger reasons such as `endpoint`, `token_threshold`, or `stop` into the replay case format
- Scoring intermediate steps against labeled ground truth
- Reconciling final todo text or optional fields in v1
- LLM-judge evaluation
- Synthetic bad-prior-todo stress tests in v1
- Automatically promoting a winning model to production

## Approved Eval Decisions

- Input per case: deterministic `reference_dt`, ordered extraction snapshots, and full expected final todo objects
- Replay state: `previous_todos` starts as `null` and is replaced at each step by that same model's output
- Step payload: transcript snapshot text, plus optional stable extraction-specific metadata such as transcript length
- Dataset contract: store only extraction-eligible replay steps, not full trace history
- Trigger semantics: do not include STT-trigger labels in the dataset contract
- Output: structured `Todo` list at each replay step; the final step result is the scored output
- Dataset: `backend/evals/incremental_extraction_quality/todo_extraction_replay_v1.json`
- Initial metric: `final_todo_count_match`
- Intermediate outputs: recorded as trace evidence or run artifacts, not scored labels

## Dataset Boundary

The incremental extraction replay eval is session-replay and extraction-only:

- source seed: recorded session artifacts and fixture pairs that can be converted into ordered extraction snapshots
- canonical eval asset: `backend/evals/incremental_extraction_quality/todo_extraction_replay_v1.json`
- case shape includes `reference_dt`, ordered replay steps, and full `expected_final_todos`
- each replay step stores only the transcript snapshot needed for extraction replay
- non-extraction trace events are excluded from the dataset contract
- trigger reasons are excluded from the dataset contract
- the dataset keeps full expected final todo objects even though the v1 evaluator only reads final todo count

Each case should represent one recorded session. Each step should represent one extraction checkpoint in that session.

## Replay Semantics

The replay runner should behave like this:

1. Initialize `previous_todos = null`
2. Run the model on the first transcript snapshot
3. Store that model output as the first step result
4. Feed that same model output into the next step as `previous_todos`
5. Continue until the final snapshot for the case
6. Score only the final replay output against the expected final todo count

The runner must not:

- inject ground-truth previous todos
- mix outputs from one model into another model's replay
- require STT-trigger labels to interpret the case

This is what preserves the trajectory effect we care about: each model must live with its own prior decisions.

## Scoring

The first version uses one deterministic metric:

- `final_todo_count_match`

How it works:

- after replaying all steps in a case, compare the predicted final todo count to the expected final todo count

Why this is enough for v1:

- it keeps the first replay eval simple and explainable
- it avoids premature text reconciliation work
- it avoids needing intermediate ground-truth labels
- it still catches drift that survives to the final user-visible result

What we are explicitly deferring:

- final todo text matching
- `due_date`, `assign_to`, `category`, `priority`, and `notification` scoring
- intermediate step scoring
- LLM-judge evaluation
- synthetic bad-prior-todo recovery tests

## Reporting Expectations

Each experiment run should make it easy to compare:

- experiment name
- model name
- provider
- prompt version
- thinking mode
- case name
- replay step count
- expected final todo count
- predicted final todo count
- pass/fail for `final_todo_count_match`
- latency and token usage where available from provider instrumentation

Per-step outputs should also be preserved in Logfire or other run artifacts so failed or suspicious runs can be inspected manually, but those intermediate outputs should not be turned into scored metrics in v1.

## Relationship To Other Eval Work

- Item 6 remains the transcript-only, from-scratch extraction eval
- Item 6.5 adds the replayed incremental extraction track where models carry their own history
- STT model evals remain a separate track and should not leak trigger semantics into this harness
- the stack comparison document in `docs/references/eval-models-and-stacks.md` remains the broader candidate inventory
- this design is the narrower source of truth for the incremental replay extraction eval track

## Open Questions After The First Run

- Which recorded sessions best expose drift or self-correction without overfitting to one scenario?
- When should we add a diagnostic sub-track for synthetic bad-prior-todo recovery tests?
- When, if ever, should this replay eval add richer final-output scoring beyond count?
- Which model setups remain stable enough under replayed carry-forward to justify production consideration?
