# Item 6.7 Design: DeepInfra Qwen 3.5 Extraction Evals

Scope: extend the extraction eval matrix with DeepInfra-hosted Qwen 3.5 models and document the local credential shape.

## Decision

Add two DeepInfra extraction experiments to the shared eval registry:

- `deepinfra_qwen35_9b_default` -> `Qwen/Qwen3.5-9B`
- `deepinfra_qwen35_4b_structured_tuned` -> `Qwen/Qwen3.5-4B`

Both should:

- use provider metadata `deepinfra`
- use DeepInfra's OpenAI-compatible base URL
- keep `prompt_version="v1"`
- keep `Qwen/Qwen3.5-9B` on provider-default settings
- use a tuned config for `Qwen/Qwen3.5-4B` with `temperature=0` and `max_tokens=1024`
- avoid re-adding `Qwen/Qwen3.5-2B` and `Qwen/Qwen3.5-0.8B` until they prove reliable on the structured extraction path

## Credential Contract

Use `DEEPINFRA_API_KEY` as the local environment variable name.

- add it to `backend/.env.example`
- read it from `backend/.env` for availability checks
- allow the eval runners to surface `missing DEEPINFRA_API_KEY` the same way they already do for other providers

## Implementation Notes

- keep the experiment registry shared between transcript-only and incremental replay evals
- route DeepInfra models through `pydantic_ai`'s OpenAI-compatible chat model/provider path
- keep Google and Mistral behavior unchanged for existing experiments
- do not change production defaults; this is an eval-matrix expansion
