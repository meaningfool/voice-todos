# Item 2.5 Design: Reliability Hardening and Test Refactor

Scope: follow-on work after Item 2. No new core product capability was added. This item hardens the existing voice-to-transcript-to-todos flow so it is safer to run, easier to reason about, and better protected by deterministic tests.

## Why this exists

Item 2 worked end-to-end, but the implementation had a few structural risks:

- session recording was always on, which meant raw mic audio, Soniox traffic, and extracted todos were written to disk by default
- the stop/finalize path could silently degrade into partial transcript extraction if Soniox was slow
- relative-date extraction asked the model to resolve "tomorrow" and "next Friday" without giving it an explicit reference date/timezone
- frontend browser resources (`WebSocket`, `MediaStream`, `MediaRecorder`, `AudioContext`, blob URLs) were not cleaned up centrally
- several tests were green without proving the real runtime path, because they duplicated logic or treated `"started"` and `"error"` as equally acceptable outcomes

Item 2.5 fixes those issues without changing the app's main interaction model.

## Goals

- Make session recording opt-in instead of always-on
- Make transcript assembly shared runtime logic instead of duplicated test logic
- Make stop/finalize failures explicit to the user instead of silent
- Give extraction a deterministic date/time reference
- Tighten model validation for due dates and reminder timestamps
- Add direct hook-level coverage for frontend lifecycle behavior
- Make default test runs deterministic and vendor-independent

## Non-goals

- No database or persistence layer
- No new UI flows beyond warning surfacing
- No real-time todo extraction while speaking
- No replacement of Soniox or Gemini

## Implemented changes

### 1. Backend settings and runtime controls

`backend/app/config.py`

- Replaced eager module-level settings instantiation with `get_settings()`
- Added `record_sessions: bool = False`
- Added `soniox_stop_timeout_seconds: float = 30.0`

This keeps the runtime behavior configurable and disables local session capture unless explicitly requested.

### 2. Typed extraction model

`backend/app/models.py`

The todo schema now validates:

- `due_date` as `date | None`
- `notification` as `datetime | None`

This prevents arbitrary strings from being treated as valid structured dates. The WebSocket layer serializes these values with `model_dump(..., mode="json")` so the frontend still receives ISO strings.

### 3. Explicit extraction context

`backend/app/extract.py`

Added `_build_extraction_input(transcript, reference_dt)` and changed `extract_todos()` to:

- accept `reference_dt: datetime | None`
- default to `datetime.now().astimezone()`
- prepend current local datetime, local date, and timezone before the transcript sent to the agent

This makes relative-date extraction grounded in a real reference point instead of leaving it implicit.

### 4. Shared transcript accumulation

New module: `backend/app/transcript_accumulator.py`

Introduced `TranscriptAccumulator` to own:

- final token accumulation
- interim token replacement
- `<fin>` handling
- final full transcript assembly

This logic is now shared between production code and replay-style tests.

### 5. WebSocket hardening

`backend/app/ws.py`

Key changes:

- invalid JSON control messages now return `{ "type": "error", "message": "Invalid control message" }`
- session recorder is only created when `record_sessions` is enabled
- start resets transcript state via `TranscriptAccumulator`
- stop waits for relay completion using the configurable timeout
- if finalize times out, the backend sends:
  - `todos` with an empty list
  - `stopped` with the transcript plus a warning
- if extraction fails, the backend still returns `stopped` and includes a warning instead of failing silently
- raw audio frames still forward correctly even when session recording is disabled

### 6. Frontend state and lifecycle refactor

`frontend/src/hooks/transcriptReducer.ts`
`frontend/src/hooks/useTranscript.ts`
`frontend/src/App.tsx`

The frontend now uses shared pure transcript helpers:

- `applyTranscriptTokens()`
- `finalizeTranscript()`
- `EMPTY_TRANSCRIPT_STATE`

`useTranscript()` was refactored to:

- keep transcript state in a ref-backed structure instead of duplicating update logic inline
- centralize teardown in a single `cleanup()` path
- clear both the 200ms mic-tail timer and the 30s stop timeout
- stop and tear down `MediaRecorder`, `MediaStream`, `AudioContext`, `AudioWorkletNode`, and `WebSocket` consistently
- revoke stale mic-recording blob URLs before starting a new recording
- revoke the final blob URL on unmount
- guard `stop()` so it only runs while actually recording
- store `warningMessage` and expose it to the UI

`App.tsx` now renders a visible warning banner when the backend reports a stop-time or extraction problem.

## Protocol changes

The existing protocol remains mostly the same, but `stopped` can now include:

```json
{
  "type": "stopped",
  "transcript": "partial or full transcript",
  "warning": "Timed out waiting for the final transcript."
}
```

This is intentionally additive. Older clients that ignore unknown fields will still work, but the current frontend consumes `warning`.

## Testing strategy after refactor

### Backend

`backend/tests/test_ws.py`

- fully mocked Soniox connection
- deterministic assertions for:
  - successful start
  - surfaced connection failures
  - invalid JSON handling
  - stop-without-start no-op behavior
  - unknown message resilience
  - ordered `todos` then `stopped`
  - timeout warning behavior
  - audio frame forwarding when recording is disabled

`backend/tests/test_replay.py`

- now replays fixtures through the production `TranscriptAccumulator` instead of a duplicated accumulation function

`backend/tests/test_extract.py`

- added deterministic tests that verify extraction receives explicit datetime context
- added a pass-through test for structured agent output
- Gemini integration tests are now opt-in via `RUN_GEMINI_INTEGRATION=1`

`backend/tests/test_soniox_integration.py`

- Soniox integration tests are now opt-in via `RUN_SONIOX_INTEGRATION=1`

### Frontend

`frontend/src/hooks/useTranscript.test.tsx`

- verifies previous blob URLs are revoked before a new recording starts
- verifies unmount cleanup closes/stops live browser resources
- verifies backend warnings become user-visible hook state

`frontend/src/hooks/transcriptReducer.test.ts`

- now tests the same helper functions used by the runtime hook

`frontend/src/App.test.tsx`

- extended to cover warning banner rendering

## Notes for future agents

- If you want session capture for debugging, explicitly enable `record_sessions`; it is intentionally off by default.
- If you want live Gemini or Soniox smoke tests, export the relevant API key plus:
  - `RUN_GEMINI_INTEGRATION=1`
  - `RUN_SONIOX_INTEGRATION=1`
- If you change transcript assembly logic again, update `TranscriptAccumulator` first and keep replay tests using the production path.
- If you add new stop-time degradation cases, prefer surfacing them through `warning` rather than silently degrading.
