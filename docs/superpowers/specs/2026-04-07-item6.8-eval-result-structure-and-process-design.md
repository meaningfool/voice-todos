# Item 6.8 Design: Eval Result Structure And Process Hardening

Scope: harden the extraction eval workflow so result storage is unambiguous, incomplete runs are obvious, timing and enrichment status are trustworthy, and new experiments can be added later without rerunning an entire baseline matrix.

## Why this exists

The first DeepInfra Qwen eval pass exposed three concrete problems in the current workflow:

- transcript-only and incremental replay runs both write into `backend/evals/extraction_quality/results/`, which makes the filesystem layout misleading
- a timestamped run directory currently stands in for too many concepts at once: one execution, one comparison set, and one place to resume or append later
- timing and token analysis depends on Logfire enrichment, but the current process does not make it explicit that `pending` means the run never completed the enrichment phase

The current system is close, but it is not yet robust enough for iterative benchmarking where a user wants to:

- establish a baseline matrix once
- add one or two new models later
- compare those new models against the same prior baseline without rerunning all experiments

## Observations From Existing Result History

Current artifacts on disk show the exact failure modes:

| Path | Suite in content | Experiments | Repeat / retries | Enrichment |
|---|---|---|---|---|
| `backend/evals/extraction_quality/results/2026-04-07T15-41-27Z/` | replay | `gemini3_flash_default` | `1 / 2` | `ok` |
| `backend/evals/extraction_quality/results/2026-04-07T15-50-43Z/` | replay | five baseline models | `5 / 2` | `ok` |
| `backend/evals/extraction_quality/results/2026-04-07T17-40-42Z/` | replay | two DeepInfra models | `5 / 2` | `pending` |
| `backend/evals/extraction_quality/results/2026-04-07T17-40-42Z-01/` | transcript-only | `deepinfra_qwen35_9b_default` | `5 / 2` | `pending` |
| `backend/evals/extraction_quality/results/2026-04-07T17-51-01Z/` | transcript-only | `deepinfra_qwen35_4b_structured_tuned` | `5 / 2` | `pending` |

Additional observations:

- the replay artifacts are not mislabeled; they clearly carry `dataset_name = "todo_extraction_replay_v1"` and include `step_results`
- the transcript-only artifacts are not replay artifacts; they clearly carry `dataset_name = "todo_extraction_v1"`
- the storage path is still wrong for replay, because the incremental runner imports `DEFAULT_RESULTS_DIR` from the extraction-quality artifact module
- the older baseline runs on disk are replay-only; there is no local transcript-only baseline matrix for the original five models
- `pending` is a transient artifact state only. In the current implementation, a normal enrichment pass should rewrite that to `ok`, `partial`, or `skipped`. A file left at `pending` means the run did not finish the enrichment phase at all

## Goals

- make the result layout unambiguous from the path alone
- separate the concept of one execution from the concept of one comparison set
- support adding new experiments later without mutating old run provenance
- make incomplete or interrupted runs obvious
- make it easy to know whether timing data is trustworthy and present
- preserve compact per-experiment artifacts as the main local inspection unit
- document a repeatable operator process in the suite READMEs

## Non-goals

- replacing Logfire as the detailed observability source
- building a general analytics UI
- changing the transcript-only or replay dataset contracts
- expanding v1 scoring beyond the current evaluator sets
- forcing every later comparison to use the same git commit if the semantic eval contract has not changed

## Primary Use Cases

### 1. Fresh baseline matrix

Run all configured experiments for one suite and keep a clean, immutable record of what happened.

### 2. Add new models later

Start from an existing baseline comparison set, run only two new models, and compare them to the baseline without rerunning the original five models.

### 3. Recover from partial runs

If one experiment stalls or a process is interrupted, rerun only the missing experiments without pretending that the original run directory was a single clean execution.

### 4. Separate transcript-only from replay

A human should be able to tell from the path alone whether an artifact belongs to `extraction_quality` or `incremental_extraction_quality`.

### 5. Trust timing only when it is actually ready

If enrichment never ran or did not complete, the artifact should say so explicitly and downstream comparison code should not silently treat missing timing as real data.

## Design Options

### Option A: Mutable run directories

Keep timestamped run directories, but allow later commands to append experiments into the same directory.

Pros:

- simple to explain at first
- one folder can hold the full comparison surface

Cons:

- a timestamped "run" no longer means one execution
- append and resume semantics get mixed together
- interrupted runs become hard to reason about
- provenance gets weaker over time

### Option B: Immutable run directories plus comparison-set manifests

Every execution creates its own immutable run directory. A separate comparison-set manifest references experiment artifacts from one or more runs, as long as they are compatible.

Pros:

- strong provenance
- easy to add new models later without mutating older runs
- partial reruns are explicit
- comparison intent becomes a first-class concept instead of an implicit directory convention

Cons:

- requires one more file type and one more concept

### Option C: Content-addressed experiment store or database

Store every experiment artifact in a canonical registry and build views or queries on top.

Pros:

- maximum flexibility
- good long-term analytics surface

Cons:

- too heavy for the current repo scale
- unnecessary complexity for the immediate workflow problem

## Recommendation

Choose Option B.

This keeps one execution immutable, which is the right primitive for debugging and provenance, while also adding a first-class comparison set that can evolve over time as new models are tested.

## Recommended Result Structure

Use suite-specific roots:

```text
backend/evals/extraction_quality/results/
  runs/
    <run_id>/
      run.json
      <experiment_id>.json
  comparison_sets/
    <set_name>.json

backend/evals/incremental_extraction_quality/results/
  runs/
    <run_id>/
      run.json
      <experiment_id>.json
  comparison_sets/
    <set_name>.json
```

### Run directory

Each CLI invocation creates a new immutable run directory under the suite that was actually executed.

Suggested `run_id` shape:

- default: UTC timestamp, for example `2026-04-07T17-51-01Z`
- optional human label via CLI, for example `2026-04-07T17-51-01Z__deepinfra-qwen-addition`

### `run.json`

This is not a duplicated case summary. It is a thin manifest for run identity and completion state.

Suggested fields:

- `suite`
- `run_id`
- `status`: `running`, `completed`, `completed_with_partial_enrichment`, `incomplete`
- `started_at`
- `finished_at`
- `dataset_name`
- `prompt_family`
- `prompt_version`
- `prompt_sha`
- `repeat`
- `task_retries`
- `max_concurrency`
- `requested_experiments`
- `completed_experiments`
- `failed_experiments`
- `artifact_paths`
- `comparison_set_name` when the run contributes to one
- `execution_context`:
  - `host_label`
  - `runner`
  - `sandboxed`

### Per-experiment artifact

Keep the current per-experiment JSON as the main local inspection unit.

Amend the contract so that:

- `enrichment_status = pending` is valid only while a run is in progress
- a finalized run must not leave artifacts at `pending`
- each case row stores `status`
- each case row stores `duration_s` after enrichment when available
- each artifact records `suite`
- each artifact records `run_id`

## Comparison Sets

Comparison sets solve the "I have a baseline and now I want to add two new models" problem.

### Concept

A comparison set is a manifest that points to compatible experiment artifacts, even when those artifacts were produced by different executions on different days.

Example:

- set name: `todo-extraction-v1-repeat5-retries2-baseline`
- first membership: the original five baseline experiments
- later membership: two DeepInfra Qwen artifacts from a later run

### Comparison set manifest

Suggested fields:

- `name`
- `suite`
- `created_at`
- `updated_at`
- `compatibility_contract`
- `members`

Suggested `compatibility_contract` fields:

- `dataset_name`
- `prompt_family`
- `prompt_version`
- `prompt_sha`
- `repeat`
- `task_retries`
- `artifact_schema_version`

Suggested `members` fields:

- `experiment_id`
- `artifact_path`
- `run_id`
- `timestamp`
- `git_commit_sha`
- `comparison_notes`

### Hard compatibility checks

To add an artifact to a comparison set, these should match exactly:

- suite
- dataset name
- prompt family
- prompt version
- prompt sha
- repeat
- task retries
- artifact schema version

### Soft compatibility warnings

These should be recorded and surfaced as warnings, not hard blockers:

- git commit sha
- max concurrency
- execution context

This allows adding new models later when the semantic eval contract is unchanged, while still warning when latency comparisons may not be apples-to-apples.

## CLI Shape

Add the following concepts to both runners:

- `--run-label <slug>`: optional human label for the immutable run directory
- `--comparison-set <name>`: attach newly produced artifacts to a named comparison set after compatibility validation
- `--create-comparison-set`: create the set if it does not exist
- `--verify-results`: fail the command if any artifact remains `pending` or if a manifest is left `incomplete`

Example flow:

```bash
cd backend
.venv/bin/python evals/extraction_quality/run.py \
  --all \
  --repeat 5 \
  --task-retries 2 \
  --comparison-set todo-extraction-v1-repeat5-retries2-baseline \
  --create-comparison-set
```

Later:

```bash
cd backend
.venv/bin/python evals/extraction_quality/run.py \
  --experiment deepinfra_qwen35_9b_default \
  --experiment deepinfra_qwen35_4b_structured_tuned \
  --repeat 5 \
  --task-retries 2 \
  --comparison-set todo-extraction-v1-repeat5-retries2-baseline
```

This produces a new immutable run directory but extends the same comparison set.

## Process Hardening

### Preflight

Before running a real comparison:

1. Run `--list-experiments`
2. Confirm the intended suite
3. Confirm provider credentials
4. Confirm Logfire write credentials are available if traces are expected
5. Confirm `LOGFIRE_READ_TOKEN` is available if timing comparison is required
6. Decide whether the command is:
   - a new baseline run
   - an additive run for an existing comparison set
   - an isolated smoke test that should not join any comparison set

### During the run

The runner should:

1. create `run.json` with `status = running`
2. write each base artifact
3. run enrichment
4. rewrite each artifact to `ok`, `partial`, or `skipped`
5. finalize `run.json`

If the process is interrupted at any point, the run manifest should remain `incomplete`.

### Post-run verification

The operator process should explicitly verify:

- no artifact in the finalized run remains at `pending`
- every experiment requested either produced an artifact or is listed in `failed_experiments`
- timing comparisons are only made from enriched artifacts with non-null durations

### Reporting rule for assistants and operators

Do not report timing comparisons from artifacts that are:

- `pending`
- `skipped`
- `partial` with null duration fields

When a run is incomplete, say that directly instead of inferring latency from token counts or wall-clock impressions.

## Migration Plan

### Phase 1: Storage clarity

- give the incremental suite its own default results root
- add `suite` and `run_id` into each artifact
- add `run.json`

### Phase 2: Comparison sets

- add comparison-set manifests
- add compatibility validation
- add CLI support for attaching new runs to an existing set

### Phase 3: Process enforcement

- add `--verify-results`
- reject finalized runs that still have `pending` artifacts
- update READMEs with the operator contract

## Success Criteria

This item is successful when:

- no replay artifact is stored under the transcript-only suite path
- a user can add new models later without rerunning the original baseline
- a timestamped run directory once again means one execution
- incomplete enrichment is obvious from both `run.json` and per-experiment artifacts
- timing comparisons are only made from finalized enriched artifacts
- the README makes the intended workflow explicit enough that a future agent does not repeat the confusion from this DeepInfra run
