# Item 6.8 Design: Align Eval Lifecycle With Pydantic Evals And Logfire

Scope: retrofit the extraction eval system so it uses Pydantic Evals concepts directly, treats Logfire as the canonical store for experiment results, and removes the local artifact and enrichment workflow as the primary source of truth.

## Why this exists

The current eval flow grew before the repo had a clear model for how Pydantic Evals and Logfire should divide responsibilities.

That led to three problems:

- local result directories are carrying too many meanings at once: one CLI invocation, one collection of model runs, one comparison surface, and one place to inspect timing later
- local JSON artifacts are pretending to be the primary result record even though the richest and most trustworthy timing, token, and trace data already lives in Logfire
- terms like `run`, `experiment`, `artifact`, and `comparison set` are currently overlapping in ways that do not line up with Pydantic's own model

Before Item 6.9 introduces a benchmark abstraction above experiments, Item 6.8 needs to make the base lifecycle and vocabulary coherent.

## Goals

- adopt Pydantic Evals terminology as the canonical language for local eval work
- define a clean lifecycle from dataset definition to experiment execution to result analysis
- make Logfire the canonical store for experiment result data
- remove the current post-run enrichment model from the normal path
- keep reproducibility by recording explicit experiment metadata for every experiment
- keep a clear distinction between one experiment and one batch execution that launches several experiments
- leave enough structure for Item 6.9 to add benchmark orchestration on top

## Non-goals

- defining the benchmark abstraction itself; that is Item 6.9
- moving datasets into Logfire managed datasets; that remains optional and should not be a dependency because the managed dataset surface is still experimental
- building a new analytics UI beyond Logfire and lightweight local tooling
- changing the transcript-only or incremental replay dataset contracts
- redesigning the v1 evaluator logic beyond what is needed to establish a clean contract boundary

## Canonical terminology

### Pydantic terms

- `Case`: one eval example with inputs, optional expected output, optional metadata, and optional case-specific evaluators
- `Dataset`: the static eval contract, meaning many cases plus dataset-level evaluators and optional report evaluators
- `Experiment`: one execution of `dataset.evaluate(...)` or `dataset.evaluate_sync(...)`
- `EvaluationReport`: the result object returned by one experiment
- `Case metadata`: static per-case context defined with the case, such as fixture source or difficulty
- `Experiment metadata`: per-experiment configuration recorded in the `metadata=` argument to `evaluate(...)`
- `Metrics`: per-case numeric measurements recorded during task execution and intended for aggregation or comparison across runs, such as duration, token count, cost, or retry count
- `Attributes`: per-case contextual values recorded during task execution to explain or filter behavior, such as route taken, cache hit, or intermediate decisions; they are queryable but are not the primary aggregated comparison surface

### Repo terms

- `Batch execution`: one CLI invocation that may launch one or many experiments
- `Benchmark`: a repo-level orchestration and comparison concept defined in Item 6.9
- `Artifact export`: an optional local export derived from an `EvaluationReport` or from Logfire query results; it is not the canonical experiment record
- `Experiment name`: the human-facing label passed to `dataset.evaluate(..., name=...)` and shown in Logfire
- `Experiment ID`: the stable machine-readable identifier for one evaluated configuration
- `Execution fingerprint`: the exact hash of the executable eval contract for one evaluated configuration

### Example mapping

If one command evaluates five model configurations against the same dataset:

- that command is one batch execution
- it launches five experiments
- it produces five evaluation reports
- Logfire stores five experiment result records with their per-case traces and metrics

The batch execution is not a Pydantic experiment and should not be named or stored as if it were one.

## Why evaluators belong to the dataset contract

The evaluator set defines what "good" means for a dataset. Changing evaluators changes the scoring contract even if the cases stay identical.

Example:

- dataset cases stay the same
- run A scores only todo count
- run B scores strict field-level todo equality

Those two experiments are not directly comparable because the definition of success changed. That is why evaluators belong to the dataset contract rather than being treated as ad hoc execution settings.

For this repo, that means the comparison surface must include not only dataset identity but also evaluator-contract identity.

## Design decisions

### 1. Logfire becomes the canonical experiment-results store

For normal eval operation, Logfire is the source of truth for:

- experiment identity and experiment metadata
- per-case outcomes and failures
- per-case and aggregate durations
- token and cost metrics when available
- traces and nested spans needed for debugging

The repo remains the source of truth for:

- dataset definitions
- evaluator code
- prompt files
- experiment configuration code
- future benchmark manifests from Item 6.9

This removes the need for local result files to pretend they are the durable system of record.

### 2. Local JSON result artifacts stop being canonical

The current `results/<timestamp>/...json` model and the later Logfire enrichment step should be retired from the normal path.

Specifically:

- do not require a per-experiment local JSON file for every experiment
- do not treat local artifacts as the canonical place for timing or token data
- do not model Logfire timing availability as a local `pending` or `ok` enrichment state
- keep local exports only as optional snapshots when an operator explicitly wants offline inspection or interchange

If export tooling is kept, it should be framed as `export experiment results` or `export benchmark comparison`, not as the primary write path of the eval system.

### 3. Logfire groups eval history by dataset, not by batch

The Logfire Evals UI is dataset-first.

Operationally, that means:

- the datasets list is a list of dataset identities that have experiment history
- clicking a dataset opens all experiment runs recorded against that dataset
- the dataset page can contain many different batches and many repeated runs of the same experiment name
- `batch_id` is useful for operator verification, but it does not define the main UI grouping

For this repo, the practical grouping rule is:

- runs are grouped together in Logfire when they were recorded against the same dataset identity
- for local datasets, that grouping is driven by the dataset presented through the SDK, which in practice means the dataset name

Because dataset grouping is broader than batch grouping:

- `batch_id` must remain a metadata field only
- the dataset page should be expected to accumulate repeated runs over time
- a benchmark must not be inferred from a dataset page alone; that will be an Item 6.9 concern

### 4. Experiment identity uses three separate layers

The repo should stop overloading one field to mean both human display and exact configuration identity.

Every tracked experiment should have:

- `experiment_name`: a short human-facing slug shown in Logfire
- `experiment_id`: a hybrid stable ID built from the readable slug plus a short hash suffix
- `execution_fingerprint`: an exact hash of the full executable evaluation contract

These layers answer different questions:

- `experiment_name`: what configuration is this, in a form a human can scan in the UI?
- `experiment_id`: what stable machine identity should queries and future benchmark tooling use?
- `execution_fingerprint`: was this the exact same executable setup as another run?

Recommended formatting:

- `experiment_name`: short readable slug, for example `gemini3_flash__minimal__todo-extraction-v1`
- `experiment_id`: readable slug plus short hash, for example `gemini3_flash__minimal__todo-extraction-v1--7c31b92a`

The short hash in `experiment_id` should be derived from the normalized comparison coordinates, not from the full execution payload. That keeps the visible experiment identity stable across reruns that change execution-only settings such as retries or concurrency.

`execution_fingerprint` is the exact hash and should be used for dedupe, exact-equality checks, and drift detection.

### 5. Real experiments require explicit experiment metadata

Every experiment must record a complete experiment metadata payload so results remain reproducible and queryable.

Required core metadata fields:

- `suite`
- `dataset_name`
- `dataset_sha`
- `evaluator_contract_sha`
- `experiment_id`
- `thinking_mode`
- `model_name`
- `prompt_family`
- `prompt_version`
- `prompt_sha`
- `execution_fingerprint`
- `batch_id`

Optional query-aid fields:

- `provider`
- `repeat`
- `task_retries`
- `max_concurrency`
- `git_commit_sha`
- `benchmark_id` once Item 6.9 exists
- `benchmark_coordinate` once Item 6.9 exists

The required metadata surface should stay deliberately small. It should capture the comparison coordinates that matter in Logfire and reserve exact execution equality for `execution_fingerprint`.

The runner must derive task behavior, `experiment_name`, core metadata, and `execution_fingerprint` from the same configuration object so the metadata cannot drift from the actual execution.

### 6. Comparison coordinates and execution fingerprint must stay distinct

The system should distinguish between:

- comparison coordinates: the settings that define which human-visible experiment variant this is
- execution fingerprint inputs: the full executable contract, including settings that matter for exact reproducibility but should not clutter the default UI

Required comparison coordinates for Item 6.8:

- `dataset_sha`
- `evaluator_contract_sha`
- `model_name`
- `thinking_mode`
- `prompt_family`
- `prompt_version`
- `prompt_sha`

These coordinates should drive:

- `experiment_name`
- the normalized payload used to derive the `experiment_id` hash suffix

`execution_fingerprint` should include all comparison coordinates plus:

- normalized `model_settings`
- `repeat`
- `task_retries`
- `max_concurrency`
- `task_contract_sha`

`batch_id`, timestamps, trace IDs, and Logfire URLs are run-instance facts and must not be folded into the comparison coordinates or `execution_fingerprint`.

### 7. `run` is demoted to batch-execution language only

The repo should stop using `run` as a synonym for `experiment`.

From this item onward:

- one experiment means one call to `dataset.evaluate(...)`
- one batch execution means one CLI command that may launch many experiments
- batch identity is captured with `batch_id`

Example:

- `python evals/extraction_quality/run.py --all`
- this is one batch execution
- if it evaluates five models, it launches five experiments
- each experiment gets its own `experiment_id`
- all five experiments share one `batch_id`

This keeps the vocabulary aligned with Pydantic while still preserving a useful local term for CLI orchestration.

### 8. Experiment completion is no longer defined by local enrichment

The current local states `pending`, `ok`, `partial`, and `skipped` only exist because the repo writes local artifacts first and then tries to merge Logfire timing back in later.

Under the new model:

- experiment execution is complete when `dataset.evaluate(...)` finishes and emits an `EvaluationReport`
- Logfire is expected to receive the experiment data as part of the normal execution path
- if a follow-up workflow needs to query Logfire immediately, the runner may poll briefly for ingestion readiness
- timing and token data should be read from Logfire directly, not copied back into a local canonical file

This removes the false idea that a finished experiment can still be "pending" because a secondary local write step has not happened yet.

### 9. Batch verification is still required

Even though Logfire becomes canonical, operators still need a deliberate verification process after a batch execution.

Preflight checks:

1. confirm the intended suite
2. run `--list-experiments`
3. confirm provider credentials
4. confirm Logfire write credentials are available
5. confirm Logfire read credentials are available if the command will immediately verify or compare results

Batch success checks:

1. every requested experiment launched
2. every launched experiment returned an `EvaluationReport`
3. every experiment recorded the required experiment metadata
4. Logfire shows the expected number of experiments for the `batch_id`
5. timing and token comparisons are made only from Logfire-backed data, not from local wall-clock impressions

### 10. Normal tracked evals should fail fast when Logfire write credentials are missing

If Logfire is the canonical results store, the normal tracked eval path cannot quietly proceed without it.

Recommended behavior:

- tracked eval commands should fail before launch when Logfire write credentials are unavailable
- read credentials are strongly recommended but only required when the command needs immediate post-run querying or verification
- if the repo still wants an ad hoc smoke-test path without Logfire, that path should be explicit and should be documented as outside the normal tracked-eval workflow

This is a sharper contract than Item 6.6, but it matches the decision to make Logfire canonical.

### 11. Reporting becomes query-first instead of enrichment-first

The default reporting story should be:

- Pydantic prints or renders an `EvaluationReport` summary at experiment time
- Logfire provides the canonical detailed drill-down and comparison surface
- local tooling can query or export from Logfire when a lightweight markdown, JSON, or CSV summary is needed later

This is simpler than the current model because there is one authoritative results system instead of a local file plus a cloud system that needs to be merged back in.

## Required runner changes

Both eval runners should move toward the same shape:

- build one shared configuration object per experiment
- derive task behavior, `experiment_name`, and experiment metadata from that object
- assign one `batch_id` per CLI invocation
- treat each selected experiment as one Pydantic experiment
- remove `--skip-logfire-enrichment` from the normal lifecycle
- stop defaulting to local result directories as the main output contract
- print or record the `batch_id` and relevant Logfire identifiers so the operator can inspect the batch later

The transcript-only and incremental replay suites should keep separate dataset definitions and separate suite identifiers, but they should share the same lifecycle model.

## Migration plan

### Phase 1: terminology and metadata alignment

- update READMEs and runner help text to use `case`, `dataset`, `experiment`, `evaluation report`, and `batch execution`
- explain that Logfire groups eval history by dataset and that `batch_id` is verification metadata, not a top-level grouping key
- add the required experiment metadata contract
- introduce the `experiment_name` / `experiment_id` / `execution_fingerprint` identity model
- add `batch_id` generation and printing
- stop using `run` to mean both batch and experiment

### Phase 2: Logfire-first execution

- require Logfire write credentials for normal tracked evals
- remove the post-run enrichment step from the normal path
- stop treating local JSON artifacts as the canonical output of the runner
- keep only optional export commands or debug snapshots if they still add value

### Phase 3: reporting and operator workflow cleanup

- update operator documentation to verify experiments in Logfire by `batch_id`
- remove `pending` or `partial` enrichment language from the normal process
- ensure any remaining local export tooling is clearly labeled as derived output

### Phase 4: prepare the surface for benchmarks

- reserve `benchmark_id` and `benchmark_coordinate` metadata fields
- make sure both suites can be queried consistently by contract metadata
- leave orchestration of multi-experiment comparison to Item 6.9

## Relationship to Item 6.9

Item 6.8 defines the canonical lifecycle and storage model for one experiment.

Item 6.9 will define how the repo groups many comparable experiments into a benchmark, determines which experiments are missing, and compares benchmark results using Logfire as the canonical store.

## Validation plan

Item 6.8 should be considered implemented only when the following checks have been run and their observations match.

### Automated integration checks

#### 1. Experiment metadata contract test

Run a small eval through the runner with a test task or fixture dataset and assert that the emitted experiment metadata contains exactly the required core fields plus any intentionally enabled optional query-aid fields.

Expected observation:

- the report metadata includes the required core keys
- the emitted `experiment_name` follows the configured human-readable slug format
- `experiment_id` follows the hybrid slug-plus-hash format
- `execution_fingerprint` is present
- local enrichment status is not part of the required contract

#### 2. Batch semantics test

Run one CLI invocation that selects two experiments and verify the runner launches two separate Pydantic experiments under one batch execution.

Expected observation:

- two experiments are executed
- both carry the same `batch_id`
- each has its own `experiment_id`
- each has its own `experiment_name`

#### 3. Tracked-mode credential gate test

Run a tracked eval command without Logfire write credentials.

Expected observation:

- the command exits before launching any experiment
- the error states that tracked evals require Logfire write credentials

#### 4. No canonical local artifact dependency test

Run a tracked eval command in the normal path and inspect the side effects.

Expected observation:

- success does not depend on writing per-experiment JSON files into `results/`
- the runner does not perform a post-run enrichment phase
- `--skip-logfire-enrichment` is removed or unused in the normal tracked path

### Live end-to-end verification

These checks should run against a dedicated Logfire dev project because they validate the actual system boundary introduced by Item 6.8.

#### 5. Transcript-only suite smoke test

Run one real transcript-only experiment against a tiny fixture dataset in tracked mode, then query Logfire by `batch_id`.

Expected observation:

- exactly one experiment appears for that `batch_id`
- the stored experiment metadata matches the local config used for the run
- per-case duration data is visible in Logfire
- trace identifiers are present and drill-down works

#### 6. Incremental replay suite smoke test

Run one real incremental replay experiment against a tiny fixture dataset in tracked mode, then query Logfire by `batch_id`.

Expected observation:

- exactly one experiment appears for that `batch_id`
- the replay suite uses the same lifecycle model as the transcript-only suite
- per-case duration data and traces are available in Logfire

### Operator-level acceptance check

#### 7. Comparison workflow sanity check

Run one batch execution that launches at least two experiments, then inspect the batch in Logfire.

Expected observation:

- the operator can find the batch by `batch_id`
- the operator can distinguish experiments by `experiment_id`
- timing and token comparisons are taken from Logfire directly rather than from local derived files

## Success criteria

This item is successful only when checks 1 through 7 above have been executed and the observed results match the expectations in the validation plan.
