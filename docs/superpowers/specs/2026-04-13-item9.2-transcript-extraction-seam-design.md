# Item 9.2 Design: Transcript And Extraction Seam Refinement

Scope: refine the backend seam above `SttSession` so transcript state and
extraction triggers are expressed in provider-feature terms rather than Soniox
token mechanics, while preserving current user-visible behavior and keeping the
frontend websocket contract unchanged.

## Why this exists

Item 9.1 proved that the provider/session seam is viable for a second STT
provider, but it also confirmed that transcript semantics above that seam are
still too Soniox-shaped.

Today the backend still assumes Soniox-style mechanics in places that should be
app-level abstractions:

- `SttToken.is_final`
- `TranscriptAccumulatorResult.final_token_count`
- extraction triggers derived from finalized-token counts
- transcript state transitions inferred from token mechanics rather than modeled
  directly

Those choices are workable for Soniox, but they are not the right seam for:

- single-stream Voxtral, which currently behaves as stable-only additive text
- future dual-delay Voxtral, which would likely expose stable plus provisional
  transcript state through a different mechanism
- future providers that expose endpoints or finalization with different event
  syntax

Item 9.2 exists to move that seam up to the app concepts we actually care
about.

## Goals

- define the app-level transcript model explicitly as:
  - `stable_text`
  - `provisional_text`
  - `full_text = stable_text + provisional_text`
- make extraction triggers depend on transcript state and optional endpoint
  observations rather than finalized-token counts
- preserve Soniox behavior
- keep the frontend websocket payload unchanged in this item
- make the seam ready for both current single-stream Voxtral and later
  provisional-text-capable providers

## Non-goals

- redesigning the frontend transcript UI
- dual-delay Voxtral
- delay tuning or eval work
- changing provider selection
- removing provider-specific syntax internally when it is still useful beneath
  the seam

## Core Design Decision

The seam above `SttSession` should be centered on transcript state, not on
provider token mechanics.

That means:

- `stable_text` and `provisional_text` remain valid app-level concepts
- providers are free to arrive at that state however they want
- the app should not require providers to expose `is_final` as the primitive
  seam abstraction

Soniox can still map naturally into this model.
Single-stream Voxtral can map to:

- `stable_text = additive transcript so far`
- `provisional_text = ""`

Later dual-delay or other providers can map to:

- `stable_text = confirmed prefix`
- `provisional_text = tentative tail`

without changing the app-level transcript model.

## Extraction Trigger Model

There are exactly three trigger paths.

### 1. Endpoint trigger

If the provider reports an endpoint event, extraction runs immediately.

This remains an optional provider capability.

### 2. Threshold trigger

If no endpoint is observed, extraction runs when the transcript has gained at
least `TOKEN_THRESHOLD` new tokens since the last extraction baseline.

Important:

- this is not "newly finalized tokens"
- this is not `final_token_count`
- this is transcript growth in the text we use for extraction

For Item 9.2, that text should be `full_text`.

Concretely:

- the app remembers the transcript snapshot used for the last successful
  extraction
- it compares the current `full_text` token count to that baseline
- when the delta reaches the threshold, extraction runs

### 3. Finalization / stop trigger

When stop/finalization happens, the app runs one last extraction only if the
final transcript differs from the transcript used for the last successful
extraction.

If the final transcript is unchanged, it must not run a redundant final
extraction.

This is the real app-level meaning that the old `final_token_count` logic was
trying to approximate.

## Phase Plan

Item 9.2 should be implemented in three phases.

## Hard Gate Rule

The phase boundaries in this item are mandatory.

**No work on Phase 2 may begin until every Phase 1 acceptance test passes.**

**No work on Phase 3 may begin until every Phase 2 acceptance test passes.**

**If a phase acceptance test is red, the phase is not complete.**

These are hard gates, not guidelines.

## Phase 1: Explicit Transcript State Abstraction

### Objective

Make transcript state explicit and provider-feature-oriented without changing
user-visible behavior.

### Required changes

- define or reshape the app-level transcript state around:
  - `stable_text`
  - `provisional_text`
  - `full_text`
- ensure Soniox and current Voxtral both map into that state cleanly
- keep the existing frontend websocket payload unchanged
- keep stop behavior unchanged apart from using the clearer transcript-state
  model internally

### Phase 1 acceptance tests

These tests are hard gates for moving to Phase 2.

1. Integration test: Soniox-style final and interim updates produce the expected
   `stable_text`, `provisional_text`, and `full_text` transitions.
2. Integration test: single-stream Voxtral-style additive stable updates produce
   `stable_text` growth with empty `provisional_text`.
3. Integration test: replacing provisional text with identical stable text does
   not change the resulting `full_text`.
4. Backend `/ws` integration test: current Soniox transcript behavior remains
   unchanged at the websocket contract level.
5. Backend `/ws` integration test: current Mistral transcript behavior remains
   unchanged at the websocket contract level.

**Phase 2 must not begin until all five Phase 1 tests pass.**

## Phase 2: Transcript-Delta-Based Extraction Triggers

### Objective

Replace finalized-token-count-driven extraction with transcript-delta-driven
extraction, while preserving endpoint behavior.

### Required changes

- remove `final_token_count` from the app-level extraction trigger logic
- track the transcript snapshot used for the last successful extraction
- trigger extraction when:
  - an endpoint is observed, or
  - `full_text` token growth since the last extraction baseline reaches the
    configured threshold
- on stop/finalization, run a final extraction only when the final transcript
  differs from the last successfully extracted transcript

### Phase 2 acceptance tests

These tests are hard gates for moving to Phase 3.

1. Integration test: endpoint observed causes immediate extraction.
2. Integration test: without endpoint support, extraction does not run before
   transcript growth reaches the configured token threshold.
3. Integration test: without endpoint support, extraction runs as soon as the
   transcript-growth threshold is crossed.
4. Integration test: stop/finalization does not run a redundant extraction when
   the final transcript matches the last successfully extracted transcript.
5. Integration test: stop/finalization does run one final extraction when the
   final transcript differs from the last successfully extracted transcript.
6. Backend `/ws` integration test: Mistral still performs live extraction
   through threshold-based transcript growth without endpoint support.

**Phase 3 must not begin until all six Phase 2 tests pass.**

## Phase 3: Provider Mapping Cleanup Above the Adapter

### Objective

Make the seam above `SttSession` explicitly feature-based and ready for future
providers, while preserving the already-passing behavior from Phases 1 and 2.

### Required changes

- remove any remaining Soniox-specific assumptions from transcript/extraction
  logic above the adapter
- keep endpoint semantics as a capability:
  - observed
  - not observed
  - unsupported
- ensure provider mappings are explicit:
  - Soniox maps into stable/provisional transcript state and endpoint support
  - current Voxtral maps into stable-only transcript state and no endpoint
    support
- keep the frontend websocket contract unchanged

### Phase 3 acceptance tests

These tests are hard gates for declaring Item 9.2 complete.

1. Backend `/ws` integration test: Soniox path still passes unchanged under the
   refined seam.
2. Backend `/ws` integration test: Mistral path still passes unchanged under the
   refined seam.
3. Backend acceptance test: configured Mistral path reaches `started`, emits
   live transcript messages, and returns a non-empty final `stopped.transcript`.
4. Shared live smoke: `scripts/stt_ws_smoke.py --provider soniox --fixture stop-the-button`
   passes.
5. Shared live smoke: `scripts/stt_ws_smoke.py --provider mistral --fixture stop-the-button`
   passes.

**Item 9.2 is not complete until all five Phase 3 tests pass.**

## Success Criteria

Item 9.2 is complete when:

- transcript state above the adapter is explicitly modeled as stable plus
  provisional text
- extraction triggers are defined in terms of endpoint observation and
  transcript growth, not finalized-token counts
- final stop extraction is gated on transcript change, not provider token
  semantics
- Soniox remains fully supported
- current single-stream Voxtral remains fully supported
- every phase hard gate has been passed in order

- an implementation plan for Item 9.2
- then implementation in phase order with hard-gate verification between phases
