# Item 7.5 Design: Logfire-Hosted Dataset Curation With Locked Benchmark Snapshots

Scope: refine Item 7's benchmark model so a Logfire hosted dataset can be the
canonical editable curation surface without breaking benchmark comparability
when the dataset changes over time.

## Problem Statement

Item 7 assumed local canonical datasets.

This item changes the dataset model:

- the canonical dataset is a mutable hosted dataset in Logfire
- benchmark execution must remain reproducible and side-by-side comparable

Without an execution lock, additive benchmark population becomes unsafe because
later entries may run against a different dataset state than earlier entries.

## Goals

- let one Logfire hosted dataset be the canonical source of truth for eval case
  inputs and expected outputs
- keep evaluator logic local in code
- make it easy to review and edit datasets in Logfire
- let a benchmark lock itself to the exact dataset state used on first run
- detect when a benchmark is stale because the hosted dataset changed after the
  benchmark was locked
- allow operators to decide whether to keep using the stale locked dataset or
  rebase and rerun the full benchmark
- avoid a manual two-step workflow where the operator must separately download a
  dataset before using it in a benchmark
- structure the rollout in phases with hard validation gates between them

## Non-goals

- hosting evaluator logic in Logfire
- defining a built-in Logfire dataset versioning system
- replacing benchmark definitions with remote-only benchmark config
- redesigning Item 7 benchmark metadata beyond what is needed for dataset
  locking
- solving long-term binary-audio hosting for STT datasets in this item

## Current Constraints

- hosted dataset CRUD exists in Logfire, but evaluator logic remains local
- documented hosted dataset version history should not be assumed to exist
- a mutable hosted dataset is useful for curation, but unsafe as the direct
  execution target for additive benchmark population
- STT examples may need to store audio by reference such as URL, fixture path,
  or object identifier rather than embedded audio bytes
- the preferred validation gate is the highest-fidelity gate that actually
  proves the intended behavior, even when that requires live Logfire or live
  provider access

## Credential And Environment Contract

Item 7.5 depends on a consistent shared-credential model across Conductor
worktrees. The operational reference is:

- `docs/references/2026-04-13-credential-storage-and-logfire-access.md`

Required contract:

- `backend/.env` is the canonical shared store for provider credentials and
  repo-managed Logfire tokens
- `backend/.logfire/logfire_credentials.json` is the canonical shared store for
  CLI-managed Logfire credentials metadata
- `.env.dev` is for non-secret local toggles only
- ad-hoc shell exports are temporary overrides only and must not be the
  intended persistent setup

For hosted dataset access specifically:

- `LOGFIRE_DATASETS_TOKEN` is the required persistent credential
- the `.logfire` write token must not be assumed to have hosted dataset scopes
- the benchmark bootstrap and later hosted dataset CRUD flows should document
  `LOGFIRE_DATASETS_TOKEN` as the expected path

## Canonical Terms

- `hosted dataset`: the mutable Logfire dataset used for review and curation
- `locked dataset snapshot`: the exact dataset payload downloaded automatically
  by the benchmark on first use and stored locally for reproducible reruns
- `dataset content hash`: hash of the exact hosted dataset payload used to
  detect whether the hosted dataset has changed since the benchmark was locked
- `stale benchmark`: a benchmark whose locked dataset hash no longer matches the
  current hosted dataset hash

Important rule:

- the hosted dataset is the canonical curation source of truth
- the locked dataset snapshot is the canonical execution input for one benchmark
  after that benchmark is first run
- the snapshot is a benchmark-owned lock artifact, not a second curation source
  of truth

## Required Behavior

### Dataset Source

- The canonical editable dataset lives in Logfire.
- Benchmark execution must not run directly against the mutable hosted dataset
  head after the benchmark has been locked once.

Before the benchmark contract can use hosted dataset IDs, the repo must support
an explicit bootstrap step that uploads the current local canonical datasets to
Logfire and records the resulting hosted dataset IDs.

### First Run Locking

On the first benchmark run, the runner must:

1. fetch the hosted dataset payload
2. compute a dataset content hash
3. write one active local lock artifact
4. persist benchmark lock metadata
5. run benchmark entries against the local lock artifact

The operator must not need a separate dataset download step.

### Local Lock Contents

The active lock artifact must include at least:

- hosted dataset identifier
- hosted dataset name when available
- hosted dataset fetch timestamp
- locked dataset hash
- hash algorithm
- case count
- exact dataset payload used for execution

The active lock artifact path is:

- `evals/locks/<benchmark_id>.json`

The lock file must itself remain a valid dataset JSON payload so the existing
eval runners can use it directly as a dataset path.

The lock metadata field name is:

- `_benchmark_lock`

The normal dataset keys like `name`, `version`, and `rows` remain intact.

### Stale Definition

`stale` means one thing only:

- the current hosted dataset content hash differs from the locked dataset hash

The following are not part of the stale definition:

- evaluator changes
- benchmark entry changes
- reporting changes
- missing benchmark entries

### Reported Benchmark State

Benchmark reporting must expose at least:

- `stale: true | false`
- `missing_entry_ids`

Missing entries are report output only. They are not a second benchmark status.

### Run Behavior When Not Stale

If the current hosted dataset hash matches the locked dataset hash:

- the benchmark run uses the active local lock artifact
- additive population is allowed

### Run Behavior When Stale

If the current hosted dataset hash differs from the locked dataset hash:

- the benchmark is stale
- the runner must not silently use the hosted dataset head
- the default behavior is warn and stop
- the operator may override with an explicit flag

### Rebase Behavior

If the operator passes `--rebase`, the runner must:

1. fetch the current hosted dataset
2. compute the new dataset content hash
3. rewrite the active local lock artifact
4. rewrite benchmark lock metadata
5. rerun the full benchmark

This item does not require a runtime-managed history of older lock files.

### STT Dataset Constraint

For future STT datasets, hosted cases should initially store audio by reference,
for example:

- fixture path
- signed URL
- object key
- application-level recording identifier

This item does not require direct hosted storage of raw audio bytes.

## Command Contract

### Benchmark Definition

- The benchmark definition must reference a hosted dataset identity instead of a
  repo-local canonical dataset path.
- The benchmark YAML field is:

```yaml
hosted_dataset: <dataset_id>
```

The intended way to obtain that dataset ID is a repo-owned bootstrap script that:

1. reads the current canonical local dataset JSON
2. creates or updates the corresponding hosted dataset in Logfire
3. prints the resulting hosted dataset ID
4. optionally writes that ID back into the benchmark YAML

### `benchmark run`

The normal benchmark run command owns stale-benchmark behavior.

Required flags:

- default stale behavior: warn and stop
- `--allow-stale`: continue using the active local lock artifact
- `--rebase`: rewrite the active local lock from the current hosted dataset and
  rerun the full benchmark

Rebase is not a separate command.

For internal tooling such as live validation scripts, the implementation may
also expose an internal benchmark-definition loader or benchmark-directory
override so temporary benchmark definitions can exercise the same run path
without mutating committed benchmark YAML files.

### `benchmark report`

Report output should include at least:

- benchmark ID
- hosted dataset identifier
- active lock path
- locked dataset hash
- current hosted dataset hash when available
- `stale: true | false`
- missing entry IDs

## Phased Rollout

The rollout should be structured as a sequence of behavior changes with hard
gates between phases.

Each phase gate must:

- validate the intended behavioral change at integration or end-to-end level
- use the highest-fidelity validation that actually proves the behavior
- never silently skip
- report one of:
  - `PASS`: expected behavior was validated
  - `FAIL`: expected behavior was exercised and did not hold
  - `WARN`: validation could not complete, with explicit diagnostic explaining
    why

The system should not add artificial test scaffolding solely to avoid live
validation. If the most realistic proof requires live Logfire or live provider
access, that is the preferred gate.

### Phase 1: Lock On First Run

Behavior:

- benchmark definition references a hosted dataset identity
- first benchmark run fetches the hosted dataset automatically
- runner writes the active local lock artifact automatically
- benchmark execution runs against the local locked snapshot

Gate:

- run a real benchmark flow against a hosted dataset
- verify the lock artifact is created automatically
- verify execution uses the locked snapshot rather than the moving hosted
  dataset head
- if validation cannot complete, surface `WARN` with diagnostic instead of a
  silent skip

### Phase 2: Detect Stale Benchmark

Behavior:

- before benchmark population, compare the current hosted dataset hash to the
  locked dataset hash
- mark the benchmark stale if the hashes differ
- surface stale status clearly in CLI and reporting output

Gate:

- change the hosted dataset after the initial lock exists
- run the stale check through the real benchmark flow
- verify stale is detected and reported
- if validation cannot complete, surface `WARN` with diagnostic instead of a
  silent skip

### Phase 3: Explicit Stale-Benchmark Actions

Behavior:

- stale benchmarks must not silently follow the hosted dataset head
- default stale behavior is warn and stop
- `--allow-stale` continues using the locked snapshot
- `--rebase` rewrites the active lock and reruns the full benchmark

Gate:

- create a stale benchmark state using a real hosted dataset change
- verify default run warns and stops
- verify `--allow-stale` uses the old locked snapshot
- verify `--rebase` rewrites the active lock and reruns the full benchmark
- if validation cannot complete, surface `WARN` with diagnostic instead of a
  silent skip

### Post-Phase Item: Logfire Environment Tagging

Behavior:

- traces are tagged so dev and prod activity can be distinguished in Logfire

Gate:

- generate traces from at least two environments or environment modes
- verify the environment distinction is visible and queryable in Logfire
- if validation cannot complete, surface `WARN` with diagnostic instead of a
  silent skip

## Operational Workflow Validation

The following workflow should be validated as an operator flow, but it is not
the same kind of product implementation as Phases 1 through 3:

1. use the app
2. inspect traces in Logfire
3. add or edit cases in the hosted dataset
4. observe the benchmark become stale when the hosted dataset changes
5. choose either `--allow-stale` or `--rebase`
6. confirm the resulting benchmark behavior matches the operator's choice

This workflow validation exists to prove the intended curation loop is usable in
practice. It does not imply that all of those steps require new application code
in Item 7.5.

## Summary

Item 7.5 makes the hosted dataset canonical for curation and the local lock
canonical for benchmark execution.

The execution contract is:

- lock on first run
- detect stale by dataset hash drift only
- stop by default when stale
- allow explicit `--allow-stale`
- allow explicit `--rebase` on the normal run command

## Explicit Addition: Durable Acceptance Tests For Phases 1 Through 3

Item 7.5 also requires explicit durable acceptance tests for the behavioral
contract introduced by Phases 1 through 3. This addition exists so the
implementation plan picks up the concrete acceptance assets that were missing or
poorly shaped in the first pass.

The governing policy remains:

- `docs/references/2026-04-13-acceptance-tests-and-verification-policy.md`

This addition is not asking for a generic cleanup task. It is asking for the
actual acceptance tests that prove the expected behavior of Phases 1, 2, and 3.

### Required Acceptance Test For Phase 1

Phase 1 must have one durable acceptance test that proves:

- a benchmark pointed at a hosted dataset creates its lock automatically on the
  first run
- benchmark execution uses the locked dataset snapshot rather than the mutable
  hosted dataset head

This test should be named by behavior, not by item number or phase number.

### Required Acceptance Test For Phase 2

Phase 2 must have one durable acceptance test that proves:

- after the hosted dataset changes, the benchmark is detected as stale
- the stale state is surfaced clearly through the real benchmark flow

This test should be named by behavior, not by item number or phase number.

### Required Acceptance Test For Phase 3

Phase 3 must have one durable acceptance test that proves:

- a stale benchmark warns and stops by default
- `--allow-stale` continues on the existing locked snapshot
- `--rebase` rewrites the lock and reruns the full benchmark

This test should be named by behavior, not by item number or phase number.

### Placement And Maintenance Requirements

These acceptance tests should:

- live where they naturally belong technically
- remain the smallest realistic proofs of the phase behaviors
- stay distinct from lower-level supporting verification

For this item, the intended maintenance action is:

  assets with behavior-named acceptance tests for the three phase behaviors

assets should therefore be treated as transitional and replaced by the new
phase-behavior acceptance tests rather than kept as the final acceptance
surface.
