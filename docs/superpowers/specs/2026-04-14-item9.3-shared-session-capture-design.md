# Item 9.3 Design: Shared Session Capture For STT Curation

Scope: make recorded STT sessions usable as a curation inbox for future STT
eval cases by recording provider-specific traces, sharing `sessions/recent/`
across Conductor worktrees, and reserving `sessions/golden/` as tracked repo
storage for promoted recordings.

## Why this exists

The current session-recording path was still Soniox-shaped in one important way:

- a recorded session always wrote `soniox.jsonl`

That was enough for Soniox replay and incremental extraction work, but it was
not enough for STT curation once Voxtral entered the system.

There was also a worktree problem:

- `sessions/recent/` lived under each worktree's repo root
- a capture made in one Conductor worktree was not automatically visible in the
  next one

Item 10 needs a cleaner operator flow:

- record locally
- review recent captures
- promote chosen recordings into tracked repo storage
- define STT dataset rows against those promoted recordings

Item 9.3 exists to create that recording and storage seam before STT benchmark
curation depends on it.

## Goals

- record `audio.pcm`, `result.json`, and a provider-specific trace file for the
  active STT provider
- make session recording work for both Soniox and Mistral through the same
  recorder seam
- share `sessions/recent/` across Conductor worktrees
- keep `sessions/recent/` ephemeral and ignored
- keep `sessions/golden/` tracked as the durable repo location for promoted STT
  recordings

## Non-goals

- automatic promotion from `sessions/recent/` into `sessions/golden/`
- deriving STT reference transcripts automatically from provider trace files
- benchmark locking
- defining STT benchmark entries
- changing the incremental extraction replay format

## Current State Summary

- `backend/app/session_recorder.py` recorded:
  - `audio.pcm`
  - `result.json`
  - `soniox.jsonl`
- `backend/app/stt_factory.py` only routed raw provider messages into the
  recorder for Soniox
- `.gitignore` already ignored `sessions/recent/`
- Conductor setup already linked shared credentials and superpowers content, but
  not session recordings

## Design Decisions

### 1. Record provider traces with provider-specific filenames

Recorded sessions should contain:

- `audio.pcm`
- `result.json`
- `<provider>.jsonl`

Examples:

- Soniox session: `soniox.jsonl`
- Mistral session: `mistral.jsonl`

This keeps the recorder useful for both:

- Soniox replay and debugging
- STT curation from either provider path

### 2. Keep the recorder seam provider-neutral

The recorder should expose one provider-neutral raw-trace write method.

Provider-specific adapters and factories can then route raw messages or events
into that method without the recorder needing Soniox-specific knowledge.

### 3. Share `sessions/recent/` across Conductor worktrees

`sessions/recent/` should be treated like other shared local state in Conductor:

- one shared root-checkout location
- symlinked into each worktree by `scripts/conductor-setup.sh`

That gives the operator one recent-session inbox across worktrees without
committing recent captures to git.

### 4. Reserve `sessions/golden/` for promoted recordings

The intended storage split is:

- `sessions/recent/`: ignored inbox of recent captures
- `sessions/golden/`: tracked repo storage for promoted recordings used by STT
  eval curation

Item 9.3 does not require an automated promotion command. It only establishes
the storage contract that Item 10 can depend on.

## Phased Rollout

The next phase must not begin until the current phase's acceptance criteria are
implemented and verified.

### Phase 1: Capture Provider-Aware Recent Sessions

Objective:

- make recorded sessions usable as a provider-neutral STT curation inbox

Behavioral delta:

- recorded sessions include a provider-specific raw trace file instead of
  assuming Soniox-only trace naming
- Soniox and Mistral both flow through the same recorder seam

Non-goals:

- worktree sharing
- tracked golden storage

Acceptance criteria:

- Starting a recorded STT session writes `audio.pcm`, `result.json`, and a
  trace file named after the active STT provider.
- The same recorder seam can capture raw trace events from Soniox and Mistral.
- Recorded session layout no longer assumes Soniox is the only provider that
  matters for STT curation.

Acceptance tests:

- `update`
  `backend/tests/test_session_recorder.py::test_session_recorder_writes_provider_specific_trace_file`
- `update`
  `backend/tests/test_stt_factory.py::test_create_stt_session_routes_soniox_messages_to_provider_recorder`
- `update`
  `backend/tests/test_stt_factory.py::test_create_stt_session_routes_mistral_events_to_provider_recorder`
- `update`
  `backend/tests/test_stt_mistral.py::test_mistral_session_records_raw_events_through_callback`

Supporting verification:

- websocket regression coverage that session recording still works in `/ws`
- Soniox recorder compatibility coverage through the existing session recorder
  tests

Phase boundary rule:

- do not treat recorded sessions as a provider-neutral STT curation inbox until
  both Soniox and Mistral can write provider traces through the shared recorder
  seam

### Phase 2: Share Recent Inbox And Track Golden Storage

Objective:

- make recent captures visible across Conductor worktrees while keeping durable
  eval recordings separate and tracked

Behavioral delta:

- rerunning Conductor setup links `sessions/recent/` to the shared root-checkout
  inbox
- `sessions/recent/` remains ignored
- `sessions/golden/` is available as tracked repo storage for promoted
  recordings

Non-goals:

- promotion automation
- STT dataset creation
- benchmark execution

Acceptance criteria:

- A recent session recorded in one Conductor worktree is visible from another
  worktree after setup runs.
- `sessions/recent/` stays out of git.
- `sessions/golden/` can be committed as durable repo data for future STT eval
  curation.

Acceptance tests:

- `shell acceptance procedure: conductor setup links the shared recent-session
  inbox`
- `shell acceptance procedure: git ignore rules keep recent ignored and golden
  tracked`

Supporting verification:

- filesystem-level checks for `sessions/golden/.gitkeep`
- setup-script regression checks for existing shared links

Phase boundary rule:

- do not build STT dataset curation on top of session capture until the recent
  inbox is shared across worktrees and golden storage is clearly tracked

## Expected Follow-On Use In Item 10

Item 9.3 prepares the recording side of the STT curation journey:

1. record into `sessions/recent/`
2. review a capture
3. promote it into `sessions/golden/<case-id>/`
4. define an STT dataset row against that promoted recording

Item 10 then builds the benchmark dataset and benchmark execution on top of that
storage contract.
